# 解析调度器（ParseScheduler）

## 1. 目的

把文本变更 → AST 更新的流程变成「防抖 + 后台解析 + 主线程通知」的非阻塞管线，保证用户打字时主线程不卡顿。

## 2. 输入 / 输出

- **输入**：`scheduleParse(const QString& source)` — 文本源（来自 Document）
- **输出**：`astReady(shared_ptr<const MarkdownAst>)` 信号，携带不可变快照

## 3. 行为契约

### [INV-1] 防抖 30ms
连续调用 `scheduleParse` 时，只在最后一次调用后的 30ms 才真正启动解析。

### [INV-2] 解析在后台线程
使用 `QThreadPool::globalInstance()` 或私有线程池执行解析任务，禁止在主线程解析。

### [INV-3] AST 必须是不可变快照
`std::make_shared<const MarkdownAst>(...)` 构造完成后对外只读。

### [INV-4] 跨线程 signal 使用 Queued 语义
`astReady` 信号必须通过 `Qt::AutoConnection`（跨线程自动变 Queued）发出。

### [INV-5] 取消过时任务
新任务到达时，已启动但未完成的旧任务应被标记为过时，完成时丢弃结果不 emit。

### [INV-6] 安全关闭
析构时必须等待所有在途任务完成或被取消，不得泄漏解析线程。

## 4. 接口

```cpp
class ParseScheduler : public QObject {
    Q_OBJECT
public:
    explicit ParseScheduler(QObject* parent = nullptr);
    ~ParseScheduler();

    // 请求解析，自动防抖 30ms
    void scheduleParse(const QString& source);

    // 立即解析（不防抖，用于强制刷新）
    void flushNow();

signals:
    // 解析完成，主线程接收
    void astReady(std::shared_ptr<const MarkdownAst> ast);

private:
    QTimer m_debounceTimer;
    QString m_pendingSource;
    std::atomic<quint64> m_generation;  // 每次新请求递增，用于丢弃过时结果
};
```

## 5. 核心算法

### 防抖流程

```
scheduleParse(source):
    m_pendingSource = source
    ++m_generation
    m_debounceTimer.start(30)   // 重置计时器

onDebounceTimeout:
    quint64 gen = m_generation.load()
    QString src = m_pendingSource
    QThreadPool::start([=] {
        auto ast = parse(src)
        if (gen == m_generation.load())  // 没有更新的请求
            emit astReady(ast)             // Queued → 主线程接收
        // 否则丢弃
    })
```

### 取消机制

通过 `m_generation` 做「代号比对」：
- 任务启动时记录当前 generation
- 任务完成时对比全局 generation
- 不一致 → 已有新任务，丢弃本次结果

## 6. 性能预算

- 防抖延迟：固定 30 ms
- 1k 行 Markdown 解析：< 10 ms
- 10k 行 Markdown 解析：< 50 ms
- 主线程 emit 到 slot 执行：< 5 ms

## 7. 验收条件

- **[T-1]** 连续打字 1 秒，只触发 1 次 `astReady`（最后一次的结果）
- **[T-2]** 大文件解析时主线程 UI 不卡顿
- **[T-3]** 快速连续修改时，过时结果不会覆盖最新结果
- **[T-4]** 应用退出时无"后台线程未退出"警告
- **[T-5]** ThreadSanitizer 运行无数据竞争警告

## 8. 已知陷阱

### 8.1 忘记处理过时结果
用户快速打字时可能同时有多个解析任务在途。如果不做 generation 比对，最后到达的可能是中间某次的旧结果，导致视觉"回退"。

### 8.2 析构时线程未退出
直接析构 ParseScheduler 而没有等待任务完成，可能让线程写入已销毁对象。必须在析构中 `QThreadPool::waitForDone()` 或标记 stop flag。

### 8.3 使用 DirectConnection
`connect(scheduler, &ParseScheduler::astReady, main, &Main::onAst, Qt::DirectConnection)` 会让 slot 在**解析线程**执行，触碰 QWidget 崩溃。必须用 AutoConnection（默认）或 QueuedConnection。
