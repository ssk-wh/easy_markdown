# 崩溃报告收集

## 1. 目的

应用崩溃时为开发者保留足够的诊断信息（minidump + 文本摘要），并在下次启动时主动告知用户上次崩溃，方便用户附带反馈给开发者。

V1 仅覆盖 Windows；Linux SIGSEGV/SIGABRT 处理留给 V2。

## 2. 输入 / 输出

- **输入**：
  - Windows `SetUnhandledExceptionFilter` 注册的 crashHandler
  - `EXCEPTION_POINTERS*` 结构体（崩溃地址、模块、线程上下文）
  - `%APPDATA%/SimpleMarkdown/crashes/` 用户目录
- **输出**：
  - `crash.dmp`：MiniDump 文件（可在 windbg / Visual Studio 加载）
  - `crash_report.txt`：人类可读文本（异常代码 / 模块 / 偏移 / 时间戳）
  - 启动后弹窗：`Crash Report Found` 提示，指引用户打开崩溃报告目录

## 3. 行为契约（不变量）

### [INV-1] 路径在启动时预解析

`crashHandler` 是 SEH 异常路径，**禁止**在内部分配堆内存或调用 Qt API（QStandardPaths 等内部有锁、可能正好持有崩溃锁）。

启动时在 `main()` **早期**（QApplication 构造**之前**）调 `prepareCrashDir()`：
- 用 `SHGetFolderPathW(CSIDL_APPDATA)` 拿到 `%APPDATA%`
- `_snwprintf_s` 拼接路径到全局静态 `wchar_t` 缓冲区 `g_crashDirW` / `g_crashDmpW` / `g_crashTxtW`
- `SHCreateDirectoryExW` 创建目录树（已存在不报错）

`crashHandler` 只用预填的 `g_crashDmpW` / `g_crashTxtW` 调 `CreateFileW`，不再做路径计算。

### [INV-2] crashHandler 的 async-signal-safety

`crashHandler` 内部**禁止**：
- `fopen` / `fprintf` / `fclose`（CRT 状态在崩溃时刻可能损坏）
- `malloc` / `free` / `new` / `delete`
- 任何 Qt API 调用
- 抛 C++ 异常

**允许**：
- Win32 `CreateFileW` / `WriteFile` / `CloseHandle`
- `MiniDumpWriteDump` (dbghelp.dll)
- `_snprintf_s` / `_snwprintf_s`（栈缓冲区，无堆分配）
- `GetModuleHandleExA` / `GetModuleFileNameA`
- `GetLocalTime`

### [INV-3] crash.dmp 路径与命名

- 路径：`%APPDATA%\SimpleMarkdown\crashes\crash.dmp`
- 命名固定为 `crash.dmp`（每次崩溃覆盖前一次）—— V1 不保留历史，避免目录无限增长
- 文本摘要文件：`crash_report.txt`（同目录，同样固定名称）
- V2 可考虑按时间戳命名 + LRU 保留最近 N 份

### [INV-4] 文本摘要内容

`crash_report.txt` 必须包含：

1. `Exception Code: 0x` 异常代码
2. `Exception Address: ` 崩溃地址
3. `Crash module: ` + `Offset: ` 模块名 + 模块内偏移（用于 windbg `ln` 命令定位）
4. `Timestamp: YYYY-MM-DD HH:MM:SS` 本地时间

不包含用户文档内容、不包含敏感数据。

### [INV-5] 启动检测与一次性弹窗

`MainWindow::maybeShowCrashReportPrompt` 在 `showEvent` 末尾被 `singleShot(500ms)` 触发，逻辑：

1. 用 `QStandardPaths::writableLocation(AppDataLocation) + "/crashes/crash.dmp"` 检查是否存在
2. 不存在 → return
3. 读 `QFileInfo::lastModified().toMSecsSinceEpoch()` 作为 dump 的 mtime
4. 读 `QSettings::value("crashes/lastSeenMtimeMs", 0)` 作为上次"已查看"标记
5. dump mtime ≤ lastSeen → 已经看过，return
6. **写回** `crashes/lastSeenMtimeMs = dumpMtime`（即便用户点 No 也算"已通知"，避免反复弹）
7. 弹 `QMessageBox::question("Crash Report Found", ..., Yes|No)` 含路径
8. 用户点 Yes → `QDesktopServices::openUrl(QUrl::fromLocalFile(crashDir))` 打开目录

### [INV-6] 隐私

- 不自动上传任何文件到外部服务器
- 不在弹窗中默认勾选"上传"
- 用户选择"打开目录"后，由用户**自主**决定要不要把 dump 发给开发者

## 4. 接口

### 4.1 main.cpp 全局静态

```cpp
static wchar_t g_crashDirW[MAX_PATH * 2];
static wchar_t g_crashDmpW[MAX_PATH * 2];
static wchar_t g_crashTxtW[MAX_PATH * 2];

static void prepareCrashDir();           // 启动时调一次
static LONG WINAPI crashHandler(EXCEPTION_POINTERS* ep);
```

### 4.2 MainWindow 新增

```cpp
void maybeShowCrashReportPrompt();
```

### 4.3 main() 启动序列

```cpp
int main(int argc, char* argv[]) {
#ifdef _WIN32
    prepareCrashDir();             // 必须在异常处理器之前
    SetUnhandledExceptionFilter(crashHandler);
#endif
    QApplication app(...);
    ...
}
```

## 5. 性能预算

| 操作 | 目标 |
|------|------|
| `prepareCrashDir`（启动一次） | < 5 ms |
| `crashHandler`（崩溃时刻） | < 200 ms（MiniDumpWriteDump 主导） |
| `maybeShowCrashReportPrompt`（启动一次） | < 1 ms（不弹窗时） |

## 6. 验收条件

- **[T-CR-1 dump 路径]** 应用启动后 `%APPDATA%\SimpleMarkdown\crashes\` 目录存在（即使没崩溃过）
- **[T-CR-2 主动崩溃产生 dump]** 在 debug 入口主动写一个 `*(int*)0 = 0` → 进程退出后 `crash.dmp` 与 `crash_report.txt` 出现在 crashes 目录
- **[T-CR-3 文本报告内容]** `crash_report.txt` 包含 `Exception Code:` `Exception Address:` `Crash module:` `Offset:` `Timestamp:` 五行
- **[T-CR-4 dump 可加载]** `crash.dmp` 能在 windbg / Visual Studio 中以"Open Crash Dump"加载，看到调用栈
- **[T-CR-5 启动弹窗一次]** 模拟一次崩溃后，下次启动 SimpleMarkdown：弹出"Crash Report Found"对话框；点 No 关闭后，**再次**重启不再弹窗
- **[T-CR-6 打开目录按钮]** 在弹窗中点 Yes，系统文件管理器打开 `%APPDATA%\SimpleMarkdown\crashes\`，用户能看到 dump 文件
- **[T-CR-7 路径异常容错]** 把 `%APPDATA%` 设为不可写（极端测试），启动不崩溃，`maybeShowCrashReportPrompt` 优雅 return
- **[T-CR-8 不上传]** 整个流程**没有**任何网络请求；`crash.dmp` 仅停留在本地
- **[T-CR-9 i18n]** 弹窗在 zh_CN / en_US 下文案完整翻译

## 7. 已知陷阱

### 7.1 SHGetFolderPathW vs SHGetKnownFolderPath

`SHGetFolderPathW` 已被 Microsoft 标记为 deprecated，但仍可用且返回非分配路径。本 Spec 选用它是因为不需要释放。`SHGetKnownFolderPath` 返回的路径需要 `CoTaskMemFree`，在崩溃时刻不安全。

### 7.2 dbghelp.dll 与 MiniDumpWriteDump

`MiniDumpWriteDump` 在 `dbghelp.dll`，已通过 `#pragma comment(lib, "dbghelp.lib")` 链接。Windows 7+ 自带，无需额外发布。

### 7.3 与 ASLR / PDB

`crash_report.txt` 的 `Offset:` 是模块内偏移（不受 ASLR 影响）。要还原源码行号需要：
- 找到对应版本的 PDB（与发布版本一致）
- 用 windbg 命令 `ln <module>+<offset>`

PDB 不打入安装包；开发者按版本号在 release artifacts 里找。

### 7.4 dump 大小

`MiniDumpNormal` 模式生成 ~1-3 MB 的 dump，主要是线程栈 + 部分模块信息。不会包含整个堆，符合"轻量"目标。如果未来需要更详细信息可改为 `MiniDumpWithDataSegs | MiniDumpWithProcessThreadData`，但 dump 会膨胀到几十 MB。

### 7.5 反复弹窗的死循环

如果 `crash.dmp` 的 mtime 总是大于 `lastSeenMtimeMs`（比如系统时间倒退），弹窗会反复出现。本 Spec 通过"启动时立即写回 lastSeenMtimeMs = dumpMtime"避免这种反复。

### 7.6 Linux 适配（V2）

V2 应：
- 注册 `signal(SIGSEGV, ...)` / `signal(SIGABRT, ...)`
- 用 `backtrace()` + `backtrace_symbols_fd()` 写到 `~/.local/share/SimpleMarkdown/crashes/crash.txt`
- 用 `breakpad` / `crashpad` 生成 minidump（实质上 dump 格式跨平台）
- 启动检测逻辑可复用 `maybeShowCrashReportPrompt`，只是 dump 路径前缀不同

V2 实施前不影响 V1 行为——`crashHandler` 仅在 `_WIN32` 编译。
