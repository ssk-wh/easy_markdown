# 构建与发布流程

## 1. 目的

定义**本地开发者从源码到可分发安装包**的命令约定：谁负责编译、谁负责打包、两者如何衔接、失败时如何终止。核心目标是让"只想编译"和"一键出安装包"两种常见场景都有对应的单一入口脚本，而不需要开发者记忆多步命令序列或手动串联脚本。

本 Spec 只约束**开发者本机**的构建与打包流程；CI 流水线的细节由 `.github/workflows/` 下的 YAML 承担，但 CI 必须与本 Spec 保持语义一致（即 CI 也应该可以通过调用同一套脚本得到相同结果，或清晰记录偏离点）。

## 2. 输入 / 输出

### 输入
- 已 clone 的源码工作树（含 submodule）
- 本机已安装的工具链：
  - Windows：MSVC、Qt 5.15、CMake ≥ 3.16、Python 3、NSIS
  - Linux：GCC/Clang、Qt 5.15 开发包、CMake ≥ 3.16、Python 3、`dpkg-dev`、`debhelper`
- `CHANGELOG.md`：首条 `## [x.y.z] - YYYY-MM-DD` 作为发布版本号来源

### 输出
- **编译产物**（`build_on_*` 产生）：
  - Windows：`build/src/app/SimpleMarkdown.exe`
  - Linux：`build/src/app/SimpleMarkdown`
- **打包产物**（`pack_on_*` 产生）：
  - Windows：`installer/dist/`（收集后的可运行目录）+ `installer/SimpleMarkdown-<version>-Setup.exe`
  - Linux：父目录下的 `simplemarkdown_<version>_*.deb`

## 3. 行为契约（不变量）

### 职责划分

- **[INV-BUILD-1]** `build_on_win.bat` / `build_on_linux.sh` **只做编译**：CMake configure + build，产出 `build/src/app/SimpleMarkdown{.exe}`。不得执行依赖收集、打包、签名或任何发布相关动作。
- **[INV-PACK-1]** `pack_on_win.bat` / `pack_on_linux.sh` **负责完整发布流程**：编译 → 依赖收集/changelog 生成 → 调用本地打包器（NSIS / dpkg-buildpackage）→ 验证产物。

### 自动触发编译

- **[INV-PACK-2]** pack 脚本**必须**在起始阶段自动调用对应平台的 `build_on_*` release 进行编译，**不得**要求用户先手动跑 build。调用发生在任何依赖收集/CHANGELOG 解析之前。
- **[INV-PACK-3]** pack 脚本对 build 阶段传入的构建模式**固定为 release**，不接受 `debug` / `clean` 等参数转发。如果开发者希望 clean 后打包，必须显式两步：先 `build_on_* clean`，再 `pack_on_*`。
- **[INV-PACK-4]** pack 脚本不依赖"用户已经提前 collect 过依赖"这一前置条件，也不依赖 `build/` 目录的预先存在；两者都应由脚本自身保证。

### 失败传播

- **[INV-PACK-5]** pack 脚本内部调用 build 失败时，**必须立即终止**并以非零退出码返回，**禁止**继续执行 collect、CHANGELOG 解析、NSIS/dpkg 等后续步骤。
- **[INV-PACK-6]** pack 脚本必须能在每个子步骤失败后立刻退出，不得静默忽略错误。Windows `.bat` 使用 `if errorlevel 1` 或 `%errorlevel%` 判断，Linux `.sh` 使用 `set -e` 配合显式退出检查。
- **[INV-PACK-7]** pack 脚本的退出码约定：
  - `0` — 全流程成功
  - `1` — build 阶段失败（语义：编译错误）
  - `2` — 依赖收集 / CHANGELOG 解析失败
  - `3` — 打包工具（NSIS / dpkg-buildpackage）失败
  - 其他非零值 — 脚本自身的前置检查失败（工具未找到等）

### 版本号来源

- **[INV-PACK-8]** 发布版本号**唯一来源**是 `CHANGELOG.md` 的首条 `## [x.y.z] - YYYY-MM-DD`。Windows NSIS 和 Linux deb 都必须从同一个 CHANGELOG 提取，不允许在脚本、nsi、control 文件中硬编码版本号。
- **[INV-PACK-9]** CHANGELOG 解析失败（文件不存在、格式错误、未找到首条条目）必须被 pack 脚本捕获并以退出码 2 终止，错误信息要指明"CHANGELOG.md 未能提取版本号"。

### 产物新鲜度

- **[INV-PACK-10]** pack 成功返回后，必须保证：
  - Windows：`installer/dist/SimpleMarkdown.exe` 的修改时间 **不早于** 本次 pack 启动时刻；`installer/SimpleMarkdown-<version>-Setup.exe` 同理。
  - Linux：生成的 `.deb` 文件时间戳不早于 pack 启动时刻。
- **[INV-PACK-11]** build 增量机制由 CMake 负责；pack 脚本对同一版源码的重复调用应利用增量编译，不强制 clean。开发者若要强制全量构建，应先手动 `build_on_* clean`。

### build 脚本既有约定（保持不变）

- **[INV-BUILD-2]** `build_on_*` 支持的参数：`release`（默认）、`debug`、`clean`；Windows 额外支持 `-- <cmake_opts>`。pack 脚本不得改变这些参数语义。
- **[INV-BUILD-3]** build 成功时返回 0，失败时返回非零（CMake configure 失败、build 失败、可执行文件未生成均视为失败）。

## 4. 接口

### 命令行接口

```
build_on_win.bat   [release|debug|clean] [-- <cmake_opts>...]
build_on_linux.sh  [release|debug|clean]

pack_on_win.bat           # 无参数，固定 release + 完整打包
pack_on_linux.sh          # 无参数，固定 release + 完整打包
```

### 退出码

| 脚本 | 退出码 | 含义 |
|------|--------|------|
| `build_on_*` | 0 | 编译成功 |
| `build_on_*` | 非 0 | CMake/编译/产物检查失败 |
| `pack_on_*` | 0 | 打包成功 |
| `pack_on_*` | 1 | build 阶段失败（继承自 build_on_*） |
| `pack_on_*` | 2 | collect/CHANGELOG 解析失败 |
| `pack_on_*` | 3 | NSIS/dpkg 打包工具失败 |
| `pack_on_*` | 其他 | 前置检查失败（工具未找到等） |

## 5. 核心算法

### pack 流程伪代码（平台无关）

```
PROCEDURE pack():
    # Step 0: 前置检查（当前工作目录、必需工具是否存在）
    verify_cwd_is_project_root()
    verify_packaging_tool_installed()  # NSIS / dpkg-buildpackage

    # Step 1: 自动编译（INV-PACK-2）
    echo "Step 1: Building release..."
    call build_on_<platform> release
    IF exit_code != 0:
        echo "[ERROR] Build failed, aborting pack."
        exit 1                            # INV-PACK-5, INV-PACK-7

    # Step 2: 平台相关的发布前准备
    IF platform == "win":
        python installer/collect_dist.py build/src/app
        IF exit_code != 0: exit 2
        APP_VER = parse_first_version_from("CHANGELOG.md")
        IF APP_VER is None: exit 2       # INV-PACK-9
    ELSE:  # linux
        python3 scripts/gen_debian_changelog.py   # 内部读 CHANGELOG.md
        IF exit_code != 0: exit 2

    # Step 3: 调用本地打包器
    IF platform == "win":
        makensis /DAPP_VERSION=$APP_VER installer/SimpleMarkdown.nsi
        IF exit_code != 0: exit 3
    ELSE:
        dpkg-buildpackage -us -uc -b
        IF exit_code != 0: exit 3

    # Step 4: 产物验证
    verify_artifact_exists_and_fresh()    # INV-PACK-10
    echo "SUCCESS: Pack completed."
    exit 0
```

### build 流程（已稳定，保持现状）

1. 解析参数 (`release` / `debug` / `clean` / `--`)
2. `clean` 分支：`rm -rf build/`，退出
3. 定位工具链（VS / Qt）
4. CMake configure（首次或 FORCE_CLEAN 时）
5. CMake build
6. 验证 `build/src/app/SimpleMarkdown{.exe}` 存在
7. 成功返回 0，失败返回非零

## 6. 性能预算

- **[PERF-1]** build 脚本对同一版源码的**无改动重跑**应 ≤ 5 秒（CMake 检查 + 快速 no-op）。
- **[PERF-2]** pack 脚本在 build 失败时的**失败检测时间** ≤ 5 秒：不允许 pack 静默跑完所有步骤才暴露问题。
- **[PERF-3]** 完整一键打包（无缓存，全量编译）的时间预算：Windows ≤ 10 分钟，Linux ≤ 5 分钟（参考值，因机器而异）。
- **[PERF-4]** 增量打包（代码无改动，仅为了重新打包）应该 ≤ 30 秒，主要开销在 collect_dist + NSIS。

## 7. 验收条件

- **[T-1] 一键打包（Windows）**：删除 `build/` 和 `installer/dist/` 后直接运行 `pack_on_win.bat`。期望：脚本自动先编译再打包，最终生成 `installer/SimpleMarkdown-<version>-Setup.exe`，无需人工先跑 `build_on_win.bat`。
- **[T-2] 编译失败立即终止（Windows）**：人为引入一处 C++ 编译错误，运行 `pack_on_win.bat`。期望：在 build 阶段就失败退出，**不产生**新的 `installer/dist/` 内容、不调用 collect_dist.py、不调用 NSIS；脚本退出码为 1。
- **[T-3] 增量打包**：先运行 `build_on_win.bat release`，立即运行 `pack_on_win.bat`。期望：pack 内部的 build 调用触发 CMake 增量检查（几乎立即完成），不重新全量编译；总耗时 ≤ PERF-4 预算。
- **[T-4] 一键打包（Linux）**：同 T-1，使用 `pack_on_linux.sh`，验证生成 `simplemarkdown_<version>_*.deb`。
- **[T-5] 编译失败立即终止（Linux）**：同 T-2，使用 `pack_on_linux.sh`，验证 build 失败后不调用 `gen_debian_changelog.py`、不调用 `dpkg-buildpackage`，退出码为 1。
- **[T-6] 退出码语义**：分别模拟 build 失败、collect 失败（故意破坏 `collect_dist.py`）、NSIS 失败（破坏 `.nsi`），验证 pack 退出码严格符合 INV-PACK-7。
- **[T-7] CHANGELOG 解析失败**：把 `CHANGELOG.md` 首条版本行改成非法格式，运行 `pack_on_win.bat`。期望：退出码为 2，错误信息明确提示 CHANGELOG 解析失败。
- **[T-8] 产物新鲜度**：成功执行 `pack_on_win.bat` 后，`installer/dist/SimpleMarkdown.exe` 与 `installer/SimpleMarkdown-<version>-Setup.exe` 的修改时间都晚于 pack 启动时刻；Linux 同理验证 `.deb` 时间戳。
- **[T-9] build 脚本独立可用**：`build_on_win.bat release` 单独运行仍按原语义工作（不触发 collect/打包），保证"只想编译"的场景不变。

## 8. 已知陷阱

- **bat 的 errorlevel 陷阱**：子调用（`call build_on_win.bat`）返回后，判断应使用 `if errorlevel 1`（意为 `>= 1`）或 `if %errorlevel% neq 0`。**不要**写 `if %errorlevel% == 1` —— CMake/NMake 在某些失败场景下可能返回 2 或其他值，会被漏判。
- **bat 的 setlocal 传播**：如果 build 脚本里用了 `setlocal`，错误码必须通过 `exit /b <code>` 显式传出；pack 脚本用 `call` 调用时才能正确接收到。现在的 `build_on_win.bat` 已经有 `setlocal` + 多处 `exit /b 1`，符合要求。
- **sh 的 `set -e` 与命令替换**：`set -e` 下，被 `$(...)` 或 `|` 包住的失败命令**不会**让外层脚本退出，必须显式 `|| exit N` 或开 `set -o pipefail`。当前 `pack_on_linux.sh` 是直接逐行命令 + `set -e`，加入 build 调用后需要确认：`./build_on_linux.sh release` 若失败，`set -e` 会让 pack 立即退出；这是安全的，无需额外写 `|| exit 1`。但为了显式起见，仍建议加 `|| exit 1` 让退出码可控（INV-PACK-7）。
- **Windows 下 python / cmake / nmake 的退出码**：这些工具在某些异常路径下可能返回非 0 非 1 的值（例如 NMake 返回 2），必须用 `if errorlevel 1` 而不是 `== 1` 判断。
- **CHANGELOG 解析的跨平台差异**：Windows 脚本用 `findstr /r`，Linux 用 `scripts/gen_debian_changelog.py`，两者对"首条版本号"的定义必须一致。当前实现均取第一条 `## [x.y.z]`，保持同步即可。
- **PROJECT_ROOT 相对路径**：pack 脚本必须先 `cd` 到脚本所在目录（当前两个 pack 脚本已经这样做了：`cd /d "%~dp0"` 和 `cd "$(dirname "$0")"`），然后才能调用 `.\build_on_win.bat` / `./build_on_linux.sh`；否则从任意目录调用时会找不到 build 脚本。
- **Windows 下相对调用 `call .\build_on_win.bat release`**：必须使用 `call` 关键字，否则控制权不会返回当前脚本，后续步骤不执行。
- **`collect_dist.py` 的输入路径**：当前 Windows pack 脚本向 `collect_dist.py` 传入 `build\src\app`。build 输出路径需与 pack 假设一致，否则会找不到 exe。若将来改动 CMake 输出目录结构，必须同步更新 pack 脚本。
- **重复编译环境变量**：pack 内部调用 build 时，MSVC 环境变量（vcvarsall）会被 build 脚本在其 `setlocal` 作用域内设置并在返回后失效，这是好事 —— pack 作用域不被污染。注意不要反过来在 pack 里提前 call vcvars，会破坏 build 脚本自身的环境探测逻辑。
