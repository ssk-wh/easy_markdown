# 系统架构

## 1. 目的

定义 SimpleMarkdown 的**模块划分、依赖关系、线程模型、数据流**，作为所有模块 Spec 的总纲。任何跨模块的重构都必须先更新本文。

## 2. 输入 / 输出

- **输入**：用户操作（键盘/鼠标/文件系统事件）
- **输出**：屏幕像素 + 磁盘文件

## 3. 行为契约（架构级不变量）

- **[INV-1] 分层依赖**：`core` 不依赖任何模块，`parser` 只依赖 `core`，`editor`/`preview` 可依赖 `core`/`parser`，`sync` 可依赖 `editor`/`preview`，`app` 依赖所有模块
- **[INV-2] 单向依赖**：严禁循环依赖；任何模块不得反向依赖 `app`
- **[INV-3] 主线程唯一绘制**：所有 QPainter 调用必须在主线程
- **[INV-4] 解析后台化**：Markdown 解析必须在后台线程，不得阻塞主线程
- **[INV-5] AST 不可变**：解析产物必须是只读快照，主线程消费时不加锁
- **[INV-6] 无外部进程**：除了标准系统 API 外，不启动任何外部进程（浏览器、转换器、渲染器）

## 4. 接口（模块边界）

### 模块清单

| 模块 | 职责 | 源码 |

| `core` | 文本存储、文档模型、主题、最近文件 | `src/core/` |

| `parser` | Markdown 解析（cmark-gfm 封装） | `src/parser/` |

## 5. 核心算法

### 5.1 线程模型

| 线程 | 职责 |
|------|------|
| **主线程（UI）** | 编辑器输入/绘制、预览绘制、布局计算、滚动同步 |
| **解析线程池** | cmark-gfm 解析、AST 构建 |
| **搜索线程** | 搜索替换的后台匹配 |
| **图片加载线程** | 本地图片异步读取与解码 |

### 5.2 数据流

```
用户按键
  │
  ▼
EditorInput (主线程)
  │
  ▼
Document::insert() → PieceTable + UndoStack (主线程)
  │
  ├─→ SyntaxHighlighter → EditorLayout → EditorPainter
  │     (即时, < 3ms)
  │
  └─→ ParseScheduler (30ms 防抖)
        │
        ▼
      [解析线程] MarkdownParser → cmark-gfm
        │
        ▼
      shared_ptr<const MarkdownAst>
        │
        ▼ emit astReady(ast) (queued connection)
        │
      PreviewLayout (主线程)
        │
        ▼
      PreviewPainter → 屏幕
```

### 5.3 会话持久化

- 主线程 idle 时把 Tab 列表 + 光标位置 + 滚动位置写入 QSettings
- 启动时读取 QSettings，恢复所有 Tab

## 7. 验收条件

- [T-1] 任何模块都可以独立编译通过（验证依赖方向）
- [T-2] `core` 只链接 `Qt::Core`，不含 `Qt::Widgets`
- [T-3] 主线程不得出现 > 16ms 的阻塞（可用 QElapsedTimer 采样）
- [T-4] 解析线程奔溃不能导致主线程奔溃
- [T-5] AST 生命周期由 shared_ptr 管理，无内存泄漏（Valgrind/ASan 验证）

## 8. 已知陷阱

- **主线程 idle 写入 QSettings**：QSettings 在 Windows 下会触发注册表 flush，频繁调用会卡顿。必须防抖或聚合写入。
- **signal-slot 跨线程必须 queued connection**：`emit astReady(ast)` 默认是 Qt::AutoConnection，如果主线程此时阻塞，信号会排队；不要用 Qt::DirectConnection 否则会在解析线程执行 slot
- **shared_ptr 跨线程**：控制块是线程安全的，但 deref 后访问 AST 数据不需要锁是因为 AST 本身 immutable
