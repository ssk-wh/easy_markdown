> ⚠ **性质**：**产品愿景级**的架构转型评估。本 plan 挑战的是 `specs/00-产品愿景.md` 的 **[INV-1] 轻量**、**[INV-4] 自包含（不依赖 WebEngine）** 这两条最核心的产品不变量，以及 `specs/10-系统架构.md` 的整个自绘栈（`模块-preview/02`、`03` + 横切关注点全集）。

> 建议阅读顺序：§1（背景）→ §3（推荐路径）→ §7（决策问题）→ §5（冲突 Spec 面）→ 其余章节。

## 1. 背景

### 1.1 用户原话
> 考虑做一个网页版本，打开应用后直接在网页上显示，这样效果就可以渲染更漂亮一些，很多比较酷炫的效果用 CSS 都比较局限。

### 1.2 诉求解读
用户想要的**不是**把 SimpleMarkdown 做成一个浏览器 SaaS，而是：

- **交付形态仍是桌面应用**：双击图标 → 出窗口 → 编辑 / 预览 Markdown
- **内部渲染用 Web 技术**：预览视觉能媲美现代笔记软件（Notion、Obsidian、Typora）
- **诉求核心是"好看"**：不是"可编辑"（那是 `plans/2026-04-14-预览区WYSIWYG直接编辑.md` 的范畴），而是**渲染表现力**

### 1.3 为什么 CSS 比 Qt 自绘更有表现力

当前 `src/preview/PreviewPainter.cpp` 纯靠 `QPainter::drawText` / `fillRect` / `drawPath`，表达能力受限于 Qt Raster 栈。CSS / SVG / Canvas / WebGL 能做到但 QPainter 不擅长或做不到的：

| 效果 | CSS 实现 | QPainter 等价方案 | 难度对比 |
|------|---------|-------------------|---------|
| **backdrop-filter（毛玻璃）** | `backdrop-filter: blur(12px)` 一行 | 截屏 → `QGraphicsBlurEffect` → 贴回，性能差且难实时 | 🔴 CSS 压倒 |
| **transform 3D（卡片翻转、视差）** | `transform: rotateY() perspective()` | 手算 3x3 矩阵、逐字形变 | 🔴 CSS 压倒 |
| **keyframe 动画 / transition** | `@keyframes` + `transition: all 0.3s` | `QPropertyAnimation` 手动绑每个属性、逐帧 repaint | 🟠 CSS 压倒 |
| **真正的阴影层（drop-shadow + filter）** | `filter: drop-shadow() blur()` 叠加 | `QGraphicsDropShadowEffect` 只单层、性能弱 | 🟠 CSS 压倒 |
| **圆角 + 背景渐变 + 多重背景** | `background: linear-gradient(), url(), radial-gradient()` | 逐层 `drawRoundedRect` + `QLinearGradient` | 🟡 QPainter 能做，CSS 更简洁 |
| **代码块语法高亮动画（token 渐入）** | CSS animation on `<span>` | 逐 token 逐帧 alpha 插值 | 🔴 CSS 压倒 |
| **数学公式（KaTeX / MathJax）** | 接入 JS 库一行代码 | 从零写 LaTeX 渲染器 | 🔴 CSS 压倒 |
| **Mermaid 流程图 / 时序图** | 接入 `mermaid.js` 一行代码 | 无实现 | 🔴 CSS 压倒 |
| **任务清单勾选的视觉反馈** | CSS `:checked` + transition | 手动 repaint + tween | 🟡 CSS 更简洁 |
| **SVG 图标 / 动态 spinner** | `<svg>` 原生 + CSS animate | QIcon + 静态 SVG，动画需自写 | 🟠 CSS 压倒 |
| **字体 ligature / variable font** | `font-feature-settings`, `font-variation-settings` | Qt 对 variable font 支持弱 | 🟠 CSS 压倒 |
| **响应式布局（flexbox / grid）** | `display: grid; gap: 1rem` | 自写 `PreviewLayout` 几百行 | 🟡 QPainter 已做，CSS 更通用 |

**综合判断**：用户要的"酷炫 CSS 效果"是真实的表现力鸿沟，不是主观偏好。但**每一项单独分析也都有 QPainter 近似方案**——真正压倒性的是**接入 JS 生态**（KaTeX / Mermaid / highlight.js / 各种现代 CSS 特性），这是 QPainter 永远追不上的社区速度。

### 1.4 产品愿景的硬冲突

`specs/00-产品愿景.md` 明确：

- **[INV-1] 轻量**：冷启动 < 0.5 秒，空载内存 < 30 MB
- **[INV-4] 自包含**：不依赖外部浏览器/WebEngine，不走 HTML 渲染
- **§8 非目标**：
  - ❌ **WebEngine 预览**：坚持 QPainter 自绘，避免 Chromium 开销
  - ❌ **为什么不做 Electron？** 违反 INV-1 和 INV-4

**也就是说**：用户这次的诉求**直接推翻**当前产品愿景的两条硬约束和两条非目标。本 plan 的核心价值**不是给出工程方案**，而是**让主线程做出产品级取舍**：愿不愿意重写 `specs/00-产品愿景.md`？

## 2. 技术路径对比

### 路径 A：Qt WebEngine 内嵌（`QWebEngineView`）

在 Qt app 内嵌一个 Chromium：预览区和/或整个 UI 都用 HTML/CSS/JS 实现，通过 `QWebChannel` 桥接 C++ 与 JS。

- **架构**：保留 Qt Application、Qt MainWindow；编辑器和预览 widget 替换为 `QWebEngineView`；C++ 侧只负责文件 IO、AST 解析、会话管理；渲染 100% 走 HTML/CSS
- **优点**
  - 保留"桌面应用"身份（仍是单个可执行，双击打开）
  - 完整 Web 渲染能力（任意 CSS + 任意 JS 库 + KaTeX + Mermaid + highlight.js 开箱即用）
  - 离线可用（HTML/CSS/JS 打包进 qrc 或随程序附带）
  - `QWebChannel` 提供的 C++↔JS 桥足以承载文件操作、主题切换、滚动同步
  - 可复用 `cmark-gfm` 的 AST → JSON → 前端渲染，或直接输出 HTML 字符串
- **致命代价**
  - **打包体积 +300MB 起**：`QtWebEngineCore.dll` + `QtWebEngineCoreProcess.exe` + `QtWebEngineResources.pak` + ICU + Chromium runtime ≈ **300-400 MB**。当前 pack_on_win 打包产物约 30-50 MB，体积 **膨胀 8-10 倍**
  - **违反 [INV-1] 轻量**：冷启动 < 0.5s 目标破产，WebEngine 进程初始化至少 200-500ms；空载内存 < 30 MB 目标破产，Chromium 多进程模型基础占用 150-300 MB
  - **违反 [INV-4] 自包含**：WebEngine 本身就是"HTML 渲染"，产品愿景 `§8 非目标` 明确禁止
  - **多进程模型复杂性**：WebEngine 是独立子进程，crash 不传染但 IPC 增加调试难度
  - **Qt 5.15 WebEngine 的 Chromium 版本较旧**（Chromium 87 左右），现代 CSS 特性（容器查询、`:has`、新 color fn）支持不全
- **适用条件**：主线程接受产品愿景**整体改写**（轻量、自包含两条 INV 废除）

### 路径 B：原生浏览器 + localhost HTTP 服务器

C++ 侧启动一个本地 HTTP server，用户默认浏览器打开 `http://localhost:<port>/` 访问。应用形态从"桌面窗口"变为"浏览器标签页 + 系统托盘常驻进程"。

- **架构**：C++ 后端只保留 HTTP server（可选 QtHttpServer / Qt 5.15 用第三方如 `cpp-httplib` / 或 Python 启动器）；前端 HTML/CSS/JS 完全独立；C++ 和 JS 通过 REST / WebSocket 通信
- **优点**
  - **零 WebEngine 依赖**：打包体积几乎不变（只加一个静态 HTTP lib 和前端 assets）
  - **[INV-1] 轻量有望保住**：取决于前端框架选型，冷启动可压在 1s 内
  - **前后端彻底解耦**：未来做真正的 Web 版 / 手机 PWA 也顺理成章
  - 前端完全用 Web 技术，CSS 表现力上限最高
  - 热重载调试比 WebEngine 简单
- **致命代价**
  - **不再是"桌面应用"**：用户必须通过浏览器访问。Windows 用户期望的"窗口 + 任务栏 + Alt+Tab 切换"体验消失
  - **多开困难**：同一文档在两个浏览器标签打开会导致数据竞争；需要额外的文档锁
  - **文件系统权限受限**：浏览器里用 `<input type="file">` 能选文件但无法"另存为到任意路径"；Chrome 有 File System Access API 但 Firefox/Safari 支持不一；**最致命：拖拽 .md 进浏览器 ≠ 能读写这个文件**，浏览器只给你文件副本，用户保存时不会写回原路径
  - **单实例 / 端口冲突**：`[INV-APP-SINGLE-INSTANCE]` 需重做，多开要抢端口
  - **浏览器选择**：用户默认浏览器可能是 IE、旧 Firefox、各种国产壳浏览器，渲染兼容性不可控
  - **离线 ≠ 无浏览器**：无网环境能跑 localhost，但用户卸载了所有浏览器就崩了
  - **关闭窗口 ≠ 关闭进程**：关浏览器标签后后端 HTTP server 仍运行，需要系统托盘 + 显式退出
  - **违反 [INV-OS-PARITY]**：Windows 和 Linux 默认浏览器配置差异大
- **适用条件**：产品定位从"桌面应用"转向"本地 Web App"（本质等于重做一个产品）

### 路径 C：Tauri / Electron 重写

用 Tauri（Rust + OS WebView，体积 ~10MB）或 Electron（Chromium bundled，体积 ~100MB）从零重构整个应用。

- **架构**：放弃 Qt / C++ 整个技术栈。Rust/Node 后端 + React/Vue/Svelte 前端
- **优点**
  - 前端 100% Web 技术、生态成熟、招聘容易
  - Tauri 体积只有 10MB 级别（复用系统 WebView：Windows 的 WebView2 / macOS 的 WKWebView / Linux 的 WebKitGTK）
  - 跨平台天然统一
  - 社区模板、脚手架丰富
- **致命代价**
  - **废掉整个 C++/Qt 代码库**：`src/` 下 100+ 个文件、40+ 个 Spec、`.claude/` 下的测试基础设施全部作废
  - **重写成本：Tauri 3-4 人月，Electron 4-6 人月**（按经验估算，含主题、会话、i18n、打包发布）
  - **失去现有投资**：
    - 高 DPI 自绘栈（INV-DPI 全族）
    - 绝对坐标递归（INV-COORD）
    - 字体中心化（INV-8/9/10）
    - 块级缓存、viewport 裁剪
    - cmark-gfm C 绑定、KSyntaxHighlighting、GoogleTest 测试集
  - **[INV-LANG]、[INV-QT]、[INV-CMAKE]、[INV-HILITE]、[INV-TEST]** 全部推翻（`specs/20-约束与不变量.md §3`）
  - **产品愿景改写 +100%**：不再是"Qt 自绘轻量编辑器"，是另一个产品
  - **Electron 违反 [INV-1]**：100MB+ bundled Chromium、空载内存 200MB+
  - **Tauri 的 WebView2 在 Windows 依赖 Edge 运行时**，没装 Edge 的 Win10 旧版本无法运行（需 bootstrap 安装）
- **适用条件**：**项目改名重启**才能走。不建议。

### 路径 D：混合——只在预览区用 WebView，编辑器保留自绘（**推荐方向**）

`QWebEngineView` 只替换 `src/preview/`，`src/editor/` 继续用当前自绘；通过 `QWebChannel` 把 Markdown AST/HTML 推给 WebView 渲染。

- **架构**：
  ```
  ┌─────────────────────────────────────────┐
  │           MainWindow (Qt)               │
  │  ┌──────────────┬──────────────────┐    │
  │  │   Editor     │  QWebEngineView  │    │
  │  │  (自绘保留)  │  (HTML 预览)     │    │
  │  └──────┬───────┴──────────┬───────┘    │
  │         │                  │            │
  │         ▼                  ▼            │
  │   Document  ──── AST ───► HTMLRenderer  │
  │                           │             │
  │                           ▼             │
  │                   QWebChannel IPC       │
  │                           │             │
  │                           ▼             │
  │                       JS Frontend       │
  │                   (KaTeX/Mermaid/hljs)  │
  └─────────────────────────────────────────┘
  ```
- **优点**
  - **预览视觉大幅提升**：CSS / JS 生态全开，Mermaid / KaTeX / 代码块动画 / 毛玻璃等全部可用
  - **编辑器继续享受自绘优势**：打字延迟 < 3ms、自绘高 DPI 栈完整保留（`INV-DPI-*`、`INV-COORD-*`、`INV-8/9/10` 全部继续有效于编辑器侧）
  - **与 8 款主题 TOML schema 兼容**：`Theme` 结构体序列化为 CSS Custom Properties 注入 WebView，8 个主题切换变成 `document.documentElement.setAttribute('data-theme', 'matcha')`
  - **与已归档 `plans/2026-04-14-预览区WYSIWYG直接编辑.md` 的路径 A（QTextDocument 换底座）是同级颠覆**，但可叠加：WYSIWYG-C 伪方案（已承诺 2-3 周落地）与本路径不冲突，先做那个再评估是否启动 Phase 1
  - **双向滚动同步依然可行**：`QWebChannel` 推 scroll event，现有 `ScrollSync` 改为 query WebView DOM 坐标
  - **INV-4 可局部松绑**：产品愿景 §8 把 "WebEngine 预览" 从非目标改为 "预览侧允许，编辑器仍自绘"
- **致命代价（但比 A/C 小）**
  - **仍需 WebEngine 依赖**（+300MB 体积、+150MB 内存）——**本路径与路径 A 的体积代价相同**，只是 UI 架构更合理
  - **[INV-1] 轻量彻底破产**：冷启动 500ms 目标达不到，空载 30MB 目标达不到
  - **双渲染栈维护成本**：自绘栈 + Web 栈两套并行，Spec 体系要新增 `模块-preview-web/` 整个分支
  - **调试链路变长**：问题可能在 C++ / IPC / JS / CSS / Chromium 任一层
  - **[INV-SEC-OFFLINE]**：WebView 加载本地 HTML 没问题，但前端若通过 CDN 引 KaTeX/Mermaid JS 会触网——必须全部 inline / vendored，体积再涨
  - **单向数据流变化**：现状 `AST → PreviewLayout → PreviewPainter`，新方案变成 `AST → HTMLSerializer → QWebChannel → JS Renderer → DOM`，步数更多
  - **选区 / 复制 / 搜索 / marking / 滚动同步** 全部要在 JS 侧重新实现一遍
  - **与 `plans/2026-04-14-预览区WYSIWYG直接编辑.md` 冲突**：如果 WYSIWYG 走路径 A（QTextDocument 底座），那边的光标/编辑栈和本 plan 的 HTML 栈是两个互斥方向，主线程必须二选一
- **工程量估算**：**3-4 个月**（MVP 级预览替换，不含结构化编辑 / WYSIWYG）

## 3. 推荐路径

**路径 D（混合：WebView 只替换预览）作为主攻方向**，前提是主线程接受 [INV-1] 和 [INV-4] 的松绑。

### 3.1 推荐理由

1. **预览是"效果展示"场景**：用户盯着看的是预览结果，Web 技术的视觉增益在此最大化
2. **编辑器是"高频输入"场景**：打字延迟 < 3ms、IME 响应、大文件滚动丝滑，自绘栈在此有量级优势，不能丢
3. **投资保护**：`src/preview/` 重写范围可控（约 40KB 代码），`src/editor/` + `src/core/` + `src/parser/` + `src/sync/` 基本保留
4. **Spec 影响可分区隔离**：`模块-preview/*` 作废改写，横切关注点（DPI / 坐标）对编辑器侧继续有效
5. **路径 D 可作为路径 A 的子集**：如果未来决定全面 Web 化（连编辑器也换），路径 D 已经铺好了 `QWebChannel` 基础设施
6. **与 WYSIWYG-C 伪方案不冲突**：`plans/2026-04-14-预览区WYSIWYG直接编辑.md` 已推荐路径 C（2-3 周落地），主线程先做 WYSIWYG-C，稳定后再决定是否启动本 plan 的 Phase 1

### 3.2 不推荐其他路径的原因

- **路径 A（WebEngine 整体）**：UI 架构不合理，编辑器不应该跑 HTML（IME/延迟/内存无优势），用 WebEngine 也应只用在预览
- **路径 B（localhost + 浏览器）**：改变应用形态，不是用户诉求（用户要"打开应用后直接在网页上显示"——本质还是想要桌面应用 + Web 渲染）；文件系统权限是死结
- **路径 C（Tauri/Electron）**：项目改名重启成本，超出本 plan 授权范围

### 3.3 与 `plans/2026-04-14-预览区WYSIWYG直接编辑.md` 的关系

| 维度 | 本 plan（网页版本 D） | WYSIWYG plan（路径 C 伪方案） |
|------|----------------------|------------------------------|
| 目标 | 预览视觉增强 | 预览内容可插入 / 跳编辑器 |
| 性质 | 架构转型 | 功能迭代 |
| 工程量 | 3-4 个月 | 2-3 周 |
| 与 [INV-1/4] 关系 | 破坏 | 不破坏 |
| 启动优先级 | 等 WYSIWYG-C 稳定后再评估 | 现在就做 |

**同时进行有风险**：`QWebChannel` 桥接的预览端和 WYSIWYG 的"插入 3 件套"是两套编辑入口，若并行会产生语义冲突（到底是谁负责 insert？谁 owns undo 栈？）。建议**串行**：先 WYSIWYG-C，再本 plan Phase 0 POC。

## 4. 实施 phase 估算（路径 D）

### Phase 0 — POC（1 周，必做；不承诺进入 Phase 1）
**目标**：一个独立分支 `feature/webview-poc`，验证最关键的 3 个假设。

- [ ] 在一个空 Qt widget 里嵌入 `QWebEngineView`，加载本地 HTML，测冷启动耗时
- [ ] 实现 `QWebChannel` 单向通道：C++ 推 Markdown 字符串 → JS 用 `marked.js` 或自研渲染器 → 显示在 WebView
- [ ] 测量打包增量：`pack_on_win.bat` 是否真会 +300MB
- [ ] 测量空载内存：WebEngine 多进程模型的 RSS
- [ ] 验证主题切换：C++ 侧改 TOML → 通过 QWebChannel 广播 CSS Variables → 即时切换

**决策出口**：Phase 0 结果若显示体积增量 / 启动时间 / 内存 任一项突破主线程可接受红线，**放弃本 plan**。

## 6. 非目标 / 放弃项

本 plan **明确不承诺**的事项：

1. **任何代码级实施**。本 plan 的产出就是 Phase 0 POC 的 go/no-go 决策。Phase 1+ 需要另外起 plan。
2. **路径 A（WebEngine 整体替换）**：UI 架构不合理，不考虑。
3. **路径 C（Tauri / Electron 重写）**：项目改名级别的投入，超出本 plan 授权。主线程若要启动，需另立项目。
4. **路径 B（localhost 浏览器形态）**：改变应用形态（桌面 → 本地 Web App），属产品定位决策，本 plan 只记录对比，不承诺实施。需产品侧另起需求评审。
5. **与 `plans/2026-04-14-预览区WYSIWYG直接编辑.md` 并行开发**：两个 plan 对预览架构的改动方向冲突，**必须串行**。本 plan 在 WYSIWYG-C 稳定前不启动 Phase 0。
6. **不承诺保留 [INV-1] 和 [INV-4]**。如果走本 plan，这两条愿景级 INV 必须松绑——这是**用户侧和产品侧的选择**，不是工程师的权限范围。

## 7. 需要主线程决策的问题（必须先回答）

### Q1（最关键，产品愿景级）：是否接受 +300MB 打包体积 / 200MB+ 空载内存，换取 Web 渲染能力？

- **接受**：`specs/00-产品愿景.md` 的 [INV-1] 松绑，Phase 0 POC 可以启动。SimpleMarkdown 从"轻量自绘编辑器"转向"Qt + Web 混合编辑器"，与当前在 README.md / 产品定位的差异化（"没有 QTextDocument、没有 QWebEngineView"）产生冲突，需要同步改 README 和对外宣传语
- **不接受**：本 plan 直接废弃，不启动 Phase 0。同时给用户一个合理回复：**当前项目定位是"极简轻量"，Web 渲染与此矛盾；如果需要完整 Web 效果，建议用 Typora / Obsidian 等竞品**

### Q2（架构级）：是否接受"预览是 Web、编辑器是原生"的双栈架构长期维护成本？

- **接受**：Spec 体系要新增 `模块-preview-web/` 分支（~5 个新 Spec），同时原 `模块-preview/*` 进入 deprecated 状态（保留一个版本以备回滚），双栈共存期 6-12 个月
- **不接受**：那就只剩路径 A（全 WebView，编辑器也换），但编辑器换 WebView 会丢失打字延迟 < 3ms 的核心优势，基本等于放弃项目独特性 → 等价于选路径 C 重写

### Q3（优先级）：本 plan 与 `plans/2026-04-14-预览区WYSIWYG直接编辑.md`（WYSIWYG-C 伪方案 2-3 周）先做哪个？

- **先 WYSIWYG-C 后本 plan**（推荐）：WYSIWYG-C 工程量小、风险低、用户价值立竿见影；稳定后再评估是否 Phase 0 POC。这样主线程还有时间观察用户对当前预览视觉的反馈强度
- **先本 plan 后 WYSIWYG**：WYSIWYG 的插入 3 件套在 WebView 架构下实现方式不同（JS 侧调文件对话框需 IPC 回到 C++），两套架构重复投入
- **只做 WYSIWYG-C，不做本 plan**：可以。对于"视觉效果不够酷炫"这个诉求，可作为"远期 backlog"封存，短期不投入

### Q4（流程）：Phase 0 POC 的决策红线是什么？

需要主线程明确可接受阈值，POC 结果才有判据：

- 冷启动时间上限：< 1s？< 2s？
- 空载内存上限：< 100MB？< 200MB？
- 打包体积增量上限：+100MB？+300MB 可接受？
- 主题切换延迟上限：< 100ms？
- 大文档（10k 行）预览首屏渲染上限：< 500ms？

建议：主线程先给一个粗略红线，POC 结果任一项越线即 no-go。

### Q5（宣传策略）：如果启动本 plan，如何向现有用户（已认可"极简轻量"定位）沟通转型？

- 路线 A：做成可选功能（启动时开关 Web Preview on/off），极简派用户保留原体验，尝鲜派用户切换体验 → 但双栈维护成本翻倍
- 路线 B：发 2.0 重大版本，明确宣传"预览升级 Web 渲染，体积/内存有所增加，换取视觉与 JS 生态"
- 路线 C：保留 1.x 分支做 LTS 维护，2.x 分支做新架构 → 两个产品实质分家
