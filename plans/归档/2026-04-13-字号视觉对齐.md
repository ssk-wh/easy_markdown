# 字号视觉对齐（编辑器 vs 预览）

## 背景

之前的字号 bug 修复（FontDefaults.h 提取 + applyFontSize 无条件调用）解决了"两边 `pointSize` 不一致"的问题——两边都是 12pt。

但用户反馈：实际看上去**编辑区的文字仍然比预览区"视觉上更大"**。

根因不是 bug，而是字体族固有差异：
- 编辑器：等宽字体（`Consolas` / 系统 FixedFont）
- 预览：比例字体（`Segoe UI`）
- 同样 12pt 下，等宽字体的 `xHeight` / `capHeight` 普遍比比例字体大 15–20%

历史上 `EditorWidget.cpp:30` 曾硬编码 `setPointSize(11)`，**很可能是某人有意做的视觉补偿**——但因为没写进 Spec，前一次"修复"把它当 bug 改回了 12pt，于是视觉差异回归。

> 这是 SDD 要解决的典型问题：没写进 Spec 的意图，迟早会被下次修复破坏。

## 范围限制

**本 Plan 只做字号视觉补偿，不做以下事项**：
- PreviewPainter 中 11 处硬编码字号的替换 → 由 `plans/2026-04-13-预览元素缩放一致性.md` 处理
- PreviewLayout 的 `setFont` 同步 `m_monoFont` + 新增 `baseFont() / monoFont()` getter → 同上 plan 处理
- `defaultMonoFont` 增加 `sizeDelta` 参数 → 同上 plan 处理

这三件事虽然 Spec 已规定，但属于"代码块/内联代码缩放无效"那个 bug 的修复范畴，不在本 Plan 内。

## 风险

- xHeight 在不同 DPI / 字体后备 (font fallback) 下可能波动，导致补偿值不稳定。缓解：限制补偿范围 ±2pt，避免极端值。
- 用户惯常的字号映射（"我喜欢编辑器 12pt"）会被打破，因为补偿后实际 pointSize 可能是 10 或 11。缓解：UI 显示"基础字号 12pt"，告诉用户基准是预览侧。
