# 编辑器 Markdown 语法染色粒度增强

## 背景

mockup `tmp/theme_mocks/mock_a_arctic_frost.html` 编辑器区对不同 Markdown 元素采用不同色彩：frontmatter 的 key / value / `---` 分隔线各自着色、`##` 标题蓝色、行内 `code` 紫色、列表 marker / 引用前缀 `>` / 链接目标 / 强调文本都有独立色。当前 `SyntaxHighlighter` 粒度偏粗，视觉层次单薄。

## 动作

- [ ] 审计当前 `SyntaxHighlighter.cpp` 已覆盖的 token 种类，列出缺失清单
- [ ] 按优先级新增/细化以下 token（每项对应一个 Theme 字段）：
    - frontmatter：`fmDelim`（`---`）/ `fmKey` / `fmValue`
    - 标题：`h1` / `h2` / `h3` / `h4` / `h5` / `h6`（级别独立色，当前可能只一色）
    - 行内代码：`inlineCode`（独立紫色，与代码块区分）
    - 代码块围栏：`fenceDelim`（` ``` ` 本身）/ `langTag`（后面的语言名）
    - 列表标记：`listMarker`（`-` / `*` / `1.`）
    - 引用前缀：`blockquotePrefix`（`>` 本身）
    - 链接：`linkText` / `linkTarget` / `linkTitle`（括号内 url 独立色）
    - 强调：`bold` / `italic` / `boldItalic` / `strikethrough`
    - 表格：`tableBorder`（`|` 与 `---`）
    - 分隔线：`hr`（独立 `---` 与 frontmatter `---` 区分由上下文决定）
- [ ] Theme 文件扩展对应字段，浅色/深色各调一份
- [ ] 与预览侧 CodeMirror-style / 已有 theme token 字段对齐，尽量复用命名
- [ ] Spec 更新：`specs/模块-editor/` 语法高亮章节列出所有 token 种类与 INV（每种 token 必须取自 Theme，不硬编码颜色）

## 验收

- 打开含 frontmatter + 多级标题 + 代码块 + 表格 + 链接的示例文件，每种元素染色可辨别
- 切换主题 token 全部刷新，无残留默认色
- 染色改动不影响编辑性能（输入延迟 < 16ms，参考性能预算）

## 风险

- token 过细会让 Theme 字段爆炸，需取舍：用户真的看得出 `h3` 和 `h4` 不同色才分，否则合并
- 增量 rehighlight 必须保持；整篇 rehighlight 可能在大文件上卡顿
- 与预览的语义 token 保持命名一致，否则主题贡献者要维护两套字典

## 备注

本 plan 保留 `draft` 状态，实施前建议先做一次用户可用性判断：如果大多数用户只在乎 h1-h3 + 代码 + 链接的区分，可裁剪掉 boldItalic / fmKey / tableBorder 等低频 token 以降低复杂度。实施顺序按验收价值排序：frontmatter → 标题级别 → 行内代码 → 链接 → 列表/引用 → 其余。
