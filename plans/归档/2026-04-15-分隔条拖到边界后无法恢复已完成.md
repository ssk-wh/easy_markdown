# Bug：编辑/预览分隔条拖到最右（或最左）后，消失的一侧无法恢复

## 现象

用户操作：
1. 拖动编辑区与预览区中间的分隔条到**最右边**
2. 预览区宽度变成 0，完全消失
3. 此时分隔条贴在窗口最右，handle 几乎不可见，无法再次抓取并拖回来
4. 左边方向（拖到最左）没测试，但预期症状相同（编辑区消失）

## 根因

`src/app/SnapSplitter.cpp` 的构造函数没有调用 `setChildrenCollapsible(false)`。
QSplitter 默认允许子区域 collapse 到 0，导致 handle 被挤到 splitter 边缘、几乎不可见。

`createTab()` 也没有给 EditorWidget / PreviewWidget 设置 `minimumWidth`，所以即便用户精确操作也会丢失一整侧。

对比：`MainWindow` 的 `m_mainSplitter` 对 TOC panel 一侧显式调用了 `setCollapsible(1, false)`，editor/preview 这一层没有同样处理。

## 修复方案

1. `SnapSplitter` 构造函数 `setChildrenCollapsible(false)` — editor/preview 都不能被拖到 0
2. `createTab` 给 editor 和 preview 分别 `setMinimumWidth(80)` — 兜底保证任一侧始终有可视空间 + handle 可抓
3. `SnapSplitterHandle` 支持双击 handle 重置到 50/50，作为用户"救急"通道（即便路径 1、2 被将来改回去，双击恢复仍然生效）

## 动作

- [ ] `src/app/SnapSplitter.cpp` 构造函数加 `setChildrenCollapsible(false)`
- [ ] `src/app/SnapSplitter.cpp` `SnapSplitterHandle::mouseDoubleClickEvent` 覆写，通知 splitter 重置等分
- [ ] `src/app/MainWindow.cpp::createTab` 给 `tab.editor` / `tab.preview` 各调 `setMinimumWidth(80)`
- [ ] `specs/模块-app/13-分隔条吸附刻度.md` 新增 `INV-SNAP-NO-COLLAPSE`（不可折叠到 0） + `INV-SNAP-DOUBLE-CLICK-RESET`（双击恢复等分）+ T 条目
- [ ] CHANGELOG 加 Fixed

## 验收

- **T-COL-1** 把 handle 拖到最右 → 预览区宽度保持 ≥ 80，分隔条始终可见可抓，松开后仍可拖回
- **T-COL-2** 把 handle 拖到最左 → 编辑区宽度保持 ≥ 80，同样可恢复
- **T-COL-3** 双击 handle → 立即恢复 50/50 布局
- **T-COL-4** 吸附刻度（25/50/75）仍然正常工作，不受 collapse 限制影响
- **T-COL-5** 极小窗口（宽度 300px 以下）resize 时不因 minimumWidth 撕裂布局

## 风险

- `setMinimumWidth(80)` 在窗口被缩到极窄时可能让 splitter 无法正确分配（两侧最小之和 > 窗口宽度）。低概率：80+80=160，窗口小于 160 时 TOC 已经先被 clamp，一般不会触发。
- 现有吸附逻辑 `onSplitterMoved` 里 `setSizes({leftSize, rightSize})` 可能计算出比 minimumWidth 小的值——Qt 自动 clamp 到 minimumWidth，不需要改吸附算法。
- 双击 handle 可能与 Qt 原生的 handle 双击行为冲突（Qt 默认无行为），但 **QSplitterHandle** 未定义双击语义，覆写安全。
