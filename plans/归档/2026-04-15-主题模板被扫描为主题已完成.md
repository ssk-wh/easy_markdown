# Bug：主题目录中的模板文件被扫描成一个可用主题

## 现象

用户操作：
1. 菜单 视图 → 主题 → 打开主题目录（首次打开，目录为空时自动注入 `template.toml` 和 `HOW_TO.md`）
2. 菜单 视图 → 主题 → 重新扫描主题
3. 主题列表里**出现了一个名为「我的主题」（template.toml 里的 `[meta].name`）的条目**
4. 用户切换到这个"主题"会应用模板里的示例颜色——不是用户的真实主题，而是我们为"引导"注入的样板

## 根因

`core::ThemeLoader::scanDirectory` / `discoverAll` 扫描 `*.toml` 全部当作主题加载。
`template.toml` 是我们在 `2026-04-15-主题目录为空时引导自定义` plan 里新注入的**引导性文件**，本身有 `[meta]` 块（为了让用户复制后即可用），结果被 loader 一视同仁当成普通主题。

这是我们前一个 plan 的回归——当时只测了"目录空 → 注入 + 弹窗"，没测"注入后再扫描会不会把模板当主题"。

## 修复方案

最干净的修复路径有三条，按优先级：

1. **文件名硬跳过（首选，最简单可靠）**：`scanDirectory` 扫到 `template.toml` 这个确切文件名时跳过。缺点：用户如果把自己的主题也命名为 `template.toml` 会被跳过——但我们在 HOW_TO.md 和模板注释里明确说"复制并改名"，叫 template 本身就是反模式。
2. **[meta] 标记字段**：给 `template.toml` 的 `[meta]` 加 `is_template = true`，loader 读到此字段时跳过该文件。缺点：需要改 ThemeLoader 解析一个新字段；HOW_TO 也要告诉用户复制后**必须删掉这行**，增加心智负担。
3. **改放子目录**：把 `template.toml` 放到 `themes/_templates/template.toml`，loader 只扫根目录。缺点：用户打开主题目录后看不到模板，违背"目录为空自动注入让用户马上看到"的原设计。

**选方案 1** — 硬跳过 `template.toml` 这个具体名字，同时在 HOW_TO.md 补一句"**必须改成其他文件名**才能被识别"（已经有类似说法，但不够强调）。

## 验收

- **T-TMPL-1** 首次打开主题目录 → 模板注入 → 点"重新扫描主题" → 菜单不出现任何模板条目
- **T-TMPL-2** 用户复制 `template.toml` 为 `my-custom.toml` 并改 `[meta].id = "my-custom"` → 重新扫描 → 新主题出现在菜单
- **T-TMPL-3** 用户把自己真实主题错误地命名为 `template.toml` → 被跳过（edge case，预期行为；HOW_TO.md 说明过）

## 风险

- 文件名匹配大小写：Windows 不敏感，Linux 敏感。我们用 `QString::compare(name, "template.toml", Qt::CaseInsensitive)` 保持一致。
- 未来如果模板系统扩展（比如多份模板），硬编码一个名字不够；但 V1 只有一份模板，不超前设计。
