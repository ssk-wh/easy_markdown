# 菜单栏与 Tab 栏视觉分割

## 1. 背景

### 用户反馈

> Tab 栏所在水平区域和应用菜单所在的水平区域背景颜色一样，视觉上融成一块。
> 建议在 menuBar 和 TabBar 之间加一条 1px 水平分割线，或用略微不同的背景色区分。

现象：QMenuBar 和 QTabBar 的水平条带粘成一整块，用户无法一眼分辨「这里是菜单 / 这里是标签」。尤其在浅色主题（Arctic Frost / Paper Mist / Sunset Haze 等）下特别明显，因为二者都是近白/近灰的浅色。

### 根因

提交 `70b7024`（「实现 Spec INV-1 唯一数据源：移除 stylesheet 硬编码分支」）之后，`MainWindow::applyTheme` 非 dark 分支中 menuBar 和 TabBar 的 `background` 均派生自同一个 `theme.editorGutterBg`：

```cpp
// src/app/MainWindow.cpp:840
const QString chromeBg = theme.editorGutterBg.name();
// 916: QMenuBar { background: %2 ... }   —— %2 = chromeBg
// 934: QTabBar { background: %1 ... }    —— %1 = chromeBg
// 935: QTabBar::tab { background: %1 ... }
```

二者都是 `chromeBg`，QMenuBar 又没有 `border-bottom`，所以浅色主题下两个条带完全融合。dark 分支（`src/app/MainWindow.cpp:861, 873`）同样都是 `#2b2b2b`，同样没有分隔线——只是深色下边缘对比本来就弱，用户容忍度高，但严格来说同样需要修复。

### Spec 覆盖

`specs/模块-app/10-菜单栏样式.md` 目前只管「菜单项间距 / 对齐 / indicator」层面，**没有定义 menuBar 与其下方控件的分割规则**。这条属于 Spec 漏洞，需要同时补 Spec 的 INV 条目和 T 条目。

Spec `specs/模块-app/12-主题插件系统.md` 的 INV-1「唯一数据源」要求 chrome 颜色由 Theme 字段派生，本次修改必须遵守——不能再写死颜色。

## 2. 方案对比

### 方案 A：QMenuBar 加 `border-bottom: 1px solid theme.editorGutterLine`

- **做法**：`QMenuBar { border-bottom: 1px solid %1; }`，颜色取 `theme.editorGutterLine`
- **优点**：改动最小，1 行 CSS；分割线颜色本身就是 Theme 主题变量，深浅主题自动跟随；符合 HTML mock 做法（所有 mock 的 `.menubar` 都有 `border-bottom: 1px solid var(--glass-line)`）
- **缺点**：会使 menuBar 高度 +1 px，可能在极端情况下影响菜单点击热区高度（验证时注意）
- **适用**：所有主题都能受益，是最通用的解法

### 方案 B：TabBar 整体背景改用 `theme.editorBg`（与 menuBar 区分）

- **做法**：把 TabBar 和 Tab `background` 从 `chromeBg` 改成 `mainBg`（即 `theme.editorBg`），让 Tab 栏贴着编辑器的颜色
- **优点**：更符合现代编辑器（VSCode / Zed）的视觉：Tab 栏"长在编辑器顶部"，而不是"挂在菜单栏下方"；Paper Mist mock 也是这种思路（`.tabbar` 用 `#FAFAF8`，略深于 menuBar 的白）
- **缺点**：需要同步调整 `QTabBar::tab:selected` 的 `background`（当前是 `tabActiveBg = theme.editorBg`，如果 TabBar 底色也改成 editorBg，选中 Tab 就失去对比度）；Tab 选中态要重新设计
- **适用**：需要一次视觉重构，改动面大

### 方案 C：TabBar 顶部加 `border-top: 1px solid theme.editorGutterLine`

- **做法**：`QTabBar { border-top: 1px solid %1; }`
- **优点**：效果等价于方案 A；如果将来想让 TabBar 额外有「浮在菜单之上」的效果，可以只在 TabBar 侧加线
- **缺点**：Qt 的 QTabBar border-top 在某些 QStyle 下会被 `QTabWidget::pane` 的 frame 覆盖，不如 QMenuBar::border-bottom 稳定；`QTabBar` 和 `QTabBar::tab` 之间的选择器优先级容易踩坑
- **适用**：只在方案 A 有副作用（如菜单热区问题）时退而求其次

### 方案 D：渐变或 box-shadow

- **做法**：用 `qlineargradient` 做浅阴影，或 `box-shadow` 做 1px 下沿投影
- **优点**：视觉最精致，符合 Liquid Glass 审美
- **缺点**：**Qt 5 QSS 不支持 box-shadow**；QMenuBar 的 background 可以用 qlineargradient 但效果有限、难调试；很容易做成「比 1px 线还难看」；不符合项目「度量优先于装饰」的风格
- **适用**：不推荐，仅作备选

### 方案 E（组合）：A + B 的轻量组合

- **做法**：menuBar 加 `border-bottom`（方案 A），同时 TabBar 背景从 `chromeBg` 换成 `panelBg`（`theme.editorGutterBg` 当前就是 panelBg，可以引入一个新的 Theme 字段 `tabBarBg` 或复用 `theme.editorBg` 的一个微调变体）
- **优点**：既有硬分割线，又有微差背景，两重保险
- **缺点**：需要为 Theme 扩字段或引入透明度调和，改动面略大；非必需

## 3. 推荐方案

**方案 A（QMenuBar 加 border-bottom）作为主方案**。

理由：

1. 改动面最小，风险可控——只在 stylesheet 字符串里加一行
2. 颜色从 `theme.editorGutterLine` 派生，完全遵守 Spec 12 INV-1「唯一数据源」，不硬编码
3. 所有 Liquid Glass HTML mock 的做法一致（`border-bottom: 1px solid var(--glass-line)`）
4. 浅色主题效果立竿见影；深色主题下 `editorGutterLine` 本身偏暗（如 `#3c3f41`），分割线在 `#2b2b2b` 背景上依然可辨
5. dark 分支的硬编码 stylesheet 可以同步加 `border-bottom: 1px solid #3c3f41`（与现有 StatusBar 的 `border-top: 1px solid #3c3f41` 对称）

**不选方案 B** 是因为它需要同步重构 Tab 选中态，出错面大；且当前视觉问题的核心是「缺分割」，而非「Tab 栏应该更像编辑器」，矫枉过正。

**不选方案 C** 是因为 Qt 5 下 QTabBar 的 border-top 在 QTabWidget 内会被 pane 遮挡，稳定性差。

**不选方案 D** 是因为 Qt 5 QSS 的能力不足，效果不值得维护成本。

**方案 E** 作为将来视觉精修阶段的备选，不在本次范围。

## 6. 风险

- **Dark 分支遗漏**：dark 分支是独立硬编码 stylesheet 字符串（`MainWindow.cpp:854-907`），非 dark 分支是 arg 化的（`MainWindow.cpp:908-972`）。改动必须**两个分支都改**，否则会出现「切到 Dark 分割线消失」。这是提交 70b7024 前后留下的结构债，本次不清偿、只同步改动。
- **QMenuBar 高度 +1 px**：加 `border-bottom: 1px` 后 QMenuBar 的 content rect 高度名义上不变但实际占位 +1 px。如果某处代码（例如自定义 titlebar、主窗口 layout 约束）硬编码了菜单栏高度，会错位 1 px。**缓解**：只在 QSS 里加，不改 C++ 代码；Qt 会自己处理；肉眼验证一次 menuBar 菜单热区没问题。
- **`theme.editorGutterLine` 在某些主题中可能太淡**：如 Paper Mist 的 `editorGutterLine` 如果非常接近背景色，分割线依然不明显。**缓解**：如果 T-13 验证失败，再补一个 plan 提出「chrome 分割线专用字段 `theme.chromeSeparator`」，本次先按 Spec 12 INV-1 用现有字段派生。
- **Spec 先改还是代码先改**：按 CLAUDE.md 黄金法则，Bug 修复如果 Spec 没覆盖，**先补 Spec 再改代码**。本 plan 的执行顺序严格遵守：动作 1 是补 Spec INV-7 + T-13/T-14，动作 2 才是改 C++。
