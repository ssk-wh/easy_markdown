> **用户追加反馈（2026-04-14）**：当前实现用了 `setCornerWidget(TopRightCorner)`，
> 按钮固定在窗口**右上角**（Sublime 风格）。但用户期望是 **Chrome/Edge 风格**——
> 「+」紧挨最后一个 Tab 右侧，随 Tab 数量水平移动，视觉上像"一个没有文字、点击会新建的 Tab"。
> 推荐方案由 A（setCornerWidget）**改为 B（自定义 QTabBar）**，继续本 plan 实施。

# Tab 栏新建按钮（「+」）

## 背景

**用户原话**：

> Tab 栏最右侧加一个「+」按钮，点击新建一个默认（未命名）文件 Tab，用户可以直接在里面编辑。
> 行为参考 VSCode / Sublime / Chrome 的新建 tab 按钮：
> - 位置：Tab 栏最后一个 Tab 右侧
> - 图标：简洁的「+」线条
> - 悬浮提示：「新建文件」/「New File」
> - 点击行为：等价于菜单 文件 → 新建（Ctrl+N）
> - 视觉跟随主题（色/hover 态）

**业界对照**：

- **VSCode**：Tab 栏右侧紧邻最后一个 tab 有一个「+」按钮；tab 很多滚动时按钮停留在右侧固定区。
- **Chrome / Edge**：「+」紧跟在最后一个 tab 后面，随 tab 数量水平移动。
- **Sublime Text**：右上角固定一个「+」，不随 tab 移动。

本项目主题 mockup（如 `tmp/theme_mocks/mock_a_arctic_frost.html` L320, L875）已出现 `.tab-new` 元素设计，样式上以圆角背景 + 轻描边 + 强调色「+」呈现，证明"固定在右侧"的形态在视觉设计上已有共识。

## 目标

- 在 `m_tabWidget` 的 Tab 栏右上角挂一个「+」按钮，点击行为等价于现有 `onNewFile()`（Ctrl+N）。
- 按钮 tooltip 走 `tr()`，保证中英 i18n 可切换。
- 按钮样式从当前 Theme 派生（hover / pressed），与周边 TabBar 视觉一致。
- 不破坏现有 Tab 右键菜单（`specs/模块-app/04-窗口焦点管理.md` INV-5）。

## 影响范围

- **修改**：
  - `src/app/MainWindow.cpp`：构造 Tab 栏 cornerWidget；stylesheet 里为该按钮添加 hover/pressed 规则（复用已有 `QTabBar QToolButton` 选择器或单独 `#tabCornerNewButton`）。
  - `src/app/MainWindow.h`：新增成员 `QToolButton* m_newTabCornerBtn = nullptr;`
  - `translations/zh_CN.ts` / `translations/en_US.ts` 等：新增 `"New File"` 翻译条目（若 TS 文件已有同词条则复用）。
- **新增**：无新增文件（SVG 图标通过 `QPainter::drawLine`/`QIcon` 生成，或直接用 Unicode「+」作为文本）。
- **Spec 变更**：
  - `specs/模块-app/04-窗口焦点管理.md`：在"Tab 栏交互"小节补一条 **INV**——"Tab 栏右上角常驻一个新建按钮，点击行为与 Ctrl+N 一致"，并追加 T 条目。

## 方案对比

### 方案 A：`QTabWidget::setCornerWidget(Qt::TopRightCorner, btn)`

把一个 `QToolButton` 挂到 `m_tabWidget` 的右上角 corner，Qt 框架负责布局。

- ✅ **极简**：5-10 行代码。
- ✅ 按钮**位置稳定**（始终在 Tab 栏右上角，不随 tab 数量抖动）。
- ✅ 与现有 stylesheet `QTabBar QToolButton { ... }`（`MainWindow.cpp:964`）兼容，主题自然跟随。
- ✅ Tab 很多出现 ◀ ▶ 滚动按钮时，`cornerWidget` 位置不受影响（Qt 的 tab scroll buttons 在 TabBar 内部，不占 corner）。
- ⚠️ 行为不完全等同 VSCode / Chrome 的「紧贴最后一个 tab」——而是类似 Sublime 的固定右上角。
- ⚠️ 高 DPI 下需确认按钮不被裁切（`setMinimumSize` + `sizeHint` 约束）。

### 方案 B：自定义 `QTabBar` 子类

重写 `paintEvent()` 额外画一个「+」区域，`mousePressEvent()` 按坐标判断落点，`tabSizeHint()` / `sizeHint()` 扩容以容纳「+」区域。

- ✅ 按钮**紧跟最后一个 tab**，与 VSCode / Chrome 视觉一致。
- ❌ 代码复杂：需处理绘制、命中测试、hover 状态、拖拽重排兼容、i18n tooltip（需重写 `event()` 响应 `QEvent::ToolTip`）。
- ❌ 易与现有功能冲突：Tab 拖拽重排（`tabMoved` 信号）、右键菜单（`tabAt()` 按索引命中）都需要显式排除 + 区域。
- ❌ 高 DPI 下自绘需手动 `devicePixelRatioF()` 处理，踩坑面积大。

### 推荐方案：**A（`setCornerWidget`）**

理由：

1. **代码改动最小**（<30 行新增代码 + 几行 stylesheet），符合 SDD 轻量推进原则。
2. Sublime 风格"固定右上角"在本项目 UI 设计语言中可接受（mock_a 也是固定位置的视觉）。
3. 已有 stylesheet（`MainWindow.cpp:964` 的 `QTabBar QToolButton` 规则）直接生效，零额外主题工作。
4. 与 Tab 拖拽、右键菜单、tab 滚动按钮完全正交，无冲突面。
5. VSCode 本身在 tab 很多时也会把「+」推到右侧固定区，退化结果一致。

## 验收条件（T 条目）

- **T1**：启动程序后，Tab 栏右上角可见一个「+」按钮；鼠标悬停 tooltip 显示 "New File"（英）或「新建文件」（中）。
- **T2**：点击「+」按钮，新增一个 `Untitled` Tab 并自动成为当前 Tab；行为与 Ctrl+N / 菜单 文件→新建 完全一致（编辑器聚焦 + 示例文本占位）。
- **T3**：Tab 数量从 1 变到 10 再到 1 的全过程中，「+」按钮始终位于 Tab 栏右上角固定区，不抖动不消失。
- **T4**：切换主题（Arctic Frost / Sunset Haze / 深色 / Paper Mist 等）时，「+」按钮背景、文字色、hover 色随主题变化，与相邻关闭按钮视觉风格一致。
- **T5**：打开 20+ 个 Tab 触发 Tab 滚动（◀ ▶ 出现）时，「+」按钮仍在右上角可点击，不被滚动按钮遮挡。
- **T6**：高 DPI（150% / 200% 缩放）下按钮不模糊不裁切，「+」居中。

## 风险

1. **`setCornerWidget` 高 DPI 偏移**：Qt 5.15 某些版本在 200% 缩放下 cornerWidget 垂直位置偏上 1-2 px；若出现，`m_newTabCornerBtn->setFixedHeight(m_tabWidget->tabBar()->sizeHint().height())` 可缓解。验证时按 CLAUDE.md 高 DPI 规范跑一遍。
2. **与 Tab 滚动按钮共存**：tab 很多时 Qt 会自动在 TabBar 左右端插入滚动按钮（`QTabBar::RightSide`）；`cornerWidget` 在 `QTabWidget` 层面，与 TabBar 的滚动按钮不冲突，但需实测确认视觉不叠放。
3. **与 Tab 右键菜单冲突**：cornerWidget 不属于 TabBar，`tabBar()->tabAt(pos)` 不会命中到它，右键菜单逻辑（`MainWindow.cpp:137-179`）天然隔离——无冲突。
4. **i18n 遗漏**：memory 中记录"新增 UI 必须走完 i18n 全链路"，已连续翻车两次（快捷键面板、欢迎页）。本 plan Step 5 单列一步防止再次遗漏。
5. **深色主题 stylesheet 缺 `QTabBar QToolButton` 规则**：`MainWindow.cpp` 的 dark 分支 L901~ 主要围绕 QDialog / 滚动条写，需要实测 cornerWidget 在深色下是否显示默认灰色；若是则补样式。
6. **图标方案选择**：纯文本「+」字符在不同字体下粗细不一；若视觉不统一，回退为嵌入 SVG QIcon。Step 2 先走文本方案做 POC，视觉不 OK 再切 SVG。
