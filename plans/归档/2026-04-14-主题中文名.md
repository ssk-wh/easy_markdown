# 内置主题 display name 中文化

## 背景

### 用户原话

当前 3 个 iOS 26 Liquid Glass 内置主题的 display name 是英文，要求改成中文：

- Arctic Frost → **冰霜蓝白**
- Sunset Haze → **琥珀桃暖**
- Midnight Aurora → **深夜极光**

菜单中看到 "Arctic Frost" 不够直观，中文用户需要脑内翻译；而项目面向中国用户为主。

### 现状盘点

- **主题文件**（`resources/themes/*.toml`）：`[meta] name = "Arctic Frost"`、`id = "arctic-frost"`、`description = "iOS 26 Liquid Glass · 冰霜蓝白"`。中文已存在于 `description` 与文件头注释中，只是 `name` 字段是英文。
- **ThemeLoader**：`loadFromFile` 直接把 `meta.name` 复制到 `Theme::name` 和 `ThemeDescriptor::displayName`。无任何 i18n/locale 处理。`discoverAll` 通过硬编码 `kBuiltinIds[]`（`"arctic-frost"` 等 id）从 `:/themes/{id}.toml` 加载，id 与 qrc 资源路径强耦合。
- **MainWindow::rebuildThemeMenu**：`m_themeMenu->addAction(d.displayName)`，即菜单项文字 = TOML 的 `meta.name` 字段，**未走 `tr()`**。
- **QSettings 持久化**：`view/themeMode` 存的是 `m_currentThemeId`，即 `"arctic-frost"` 等 id；与 `name` 无关。只改 `name` 不影响用户设置。
- **light / dark**：`meta.name = "Light"` / `"Dark"`，目前菜单显示英文原文（未 tr）。用户本次需求未要求改，但为保持三款新主题与两款旧主题的一致性（要么全中文、要么全英文），本 plan 需在方案中回答该问题。

### Q1：改哪个字段？

**只改 `meta.name`**。理由：

1. `meta.id` 是主题的唯一标识，与 qrc 资源路径（`:/themes/arctic-frost.toml`）、文件名、`kBuiltinIds[]` 三处强绑定；改 id 需要同时改文件名、qrc 条目、C++ 硬编码数组，并为老用户 QSettings 写 id 迁移 map，成本高、风险大。
2. `meta.name` 只作 display name 使用（菜单标签），改它**零破坏**。
3. 用户需求本身就是 display name，不是 id。

### Q2：是否做多语言（i18n）？

项目 i18n 现状（见 `specs/横切关注点/60-国际化.md`）：

- **INV-2**：源字符串统一用英文，禁止对中文字符串做 `tr()`。
- 当前 `displayName` 走的是 `addAction(d.displayName)`，**没有 `tr()`**。即使包了 `tr()`，源串还是 TOML 里读出来的动态字符串，lupdate 也提不出翻译条目。
- 正确 i18n 要么需要：(a) 在 TOML 里增加 `name_zh_CN` / `name_en_US` 字段 + ThemeLoader 按 QLocale 选择；(b) 在代码里为内置主题硬编码 `QHash<id, tr()>` 映射表（与"数据驱动、TOML 是唯一真相"的设计原则冲突）。

三个候选方案见下方对比。

## 方案对比

| 维度 | 方案 A：TOML `name` 直接改中文 | 方案 B：TOML 加 `name_zh_CN` + Loader 按 locale 选 | 方案 C：C++ 侧硬编码 tr() 映射表 |
|------|------------------------------|-------------------------------------------------|-----------------------------|
| 改动面 | 3 个 TOML 的 1 行 | 5 个 TOML（含 Light/Dark）各加 1 行 + ThemeLoader 逻辑 + Spec INV | MainWindow::rebuildThemeMenu + 翻译资源 + 映射表 |
| 英文 locale 观感 | 菜单显示中文名（外国用户看不懂） | 中文 locale 显示中文、英文 locale 显示英文 | 同方案 B |
| 用户自定义主题 | 不受影响（用户写啥显示啥） | 用户主题若只写 `name` 照常工作，可选写 `name_zh_CN` | 用户主题仍走 `name`，无 i18n（二等公民） |
| 与 60-国际化 INV-2 冲突 | 不冲突（TOML 是数据而非源码字符串，不经 tr） | 不冲突（走数据查表而非 tr） | **冲突**（tr 中文源串违反 INV-2） |
| QSettings 兼容性 | 完全兼容（id 未变） | 完全兼容 | 完全兼容 |
| 工作量 | 极小（< 5 行） | 小（< 30 行 + spec 更新） | 中（翻译链路 + 映射） |
| 可扩展性 | 差（增语言需改 TOML） | 好（后续加 `name_ja_JP` 等） | 中（所有语言需走 .ts） |
| pros | 最简、零架构变动 | 真 i18n、数据驱动、未来可扩 | 走标准 Qt i18n 链路 |
| cons | 英文 UI 下也是中文名 | 需小改 Loader + Spec | 违反 INV-2；与 TOML-唯一-真相 设计原则冲突 |

## 推荐方案

**方案 A（TOML `name` 直接改中文），并明确放弃 Light/Dark 的同步中文化。**

理由：

1. **用户群体**：项目主要面向中文用户，英文 UI 是极少数边缘场景。显示"冰霜蓝白"对中文用户体验显著提升，对英文用户也只是看不懂一个词（类比：中文版 Windows 里的日式软件菜单也常见日文名，用户可接受）。
2. **最小改动原则**：方案 A 只改 3 行 TOML，零代码、零 Spec 改动，可完全在 30 分钟内落地。
3. **不过度工程化**：方案 B 的 `name_zh_CN` 在 V1 阶段没有真实需求（项目只支持中文+英文；英文用户通过 description 可理解）。若未来真的需要切 locale 显示不同名字，可在 V2 无痛升级到方案 B（追加字段、旧 TOML 仍兼容）。
4. **保持 Light/Dark 英文**：这两个是行业通用词汇（浅色/深色），改成中文反而显得本土化过头、与"Follow System" 等 tr 菜单项风格不一致（"跟随系统 / Light / Dark" 混搭读起来还行；但改成"跟随系统 / 浅色 / 深色" 又与三款主题的"中文雅名"风格不搭）。三个新主题因其艺术化命名，用中文更合适。
5. **演进路径清晰**：如未来方案 B 被采纳，本次只改 `name` 的改动完全是其第一步，无需回滚。

### 与 Spec 兼容性

- `specs/模块-app/12-主题插件系统.md`：未规定 `meta.name` 语种，仅规定字段类型为 string。方案 A 不违反任何 INV。
- `specs/横切关注点/60-国际化.md` INV-2：「**源字符串**统一用英文」仅约束 C++ 源码中的 `tr()` 调用；TOML 数据文件中的字符串不属于"源字符串"范畴，不在 INV-2 约束内（与现有 `description` 字段已是中文的事实一致）。

## 验收

- **[T-1]** 首次启动（QSettings 为空）应用后，主题菜单显示 3 条中文主题名「冰霜蓝白」「琥珀桃暖」「深夜极光」，Light / Dark 保持英文，排序按 `discoverAll` 返回顺序（light、dark、arctic-frost、sunset-haze、midnight-aurora）。
- **[T-2]** **旧版本升级场景**：人为把 `%APPDATA%/SimpleMarkdown` 下的 QSettings `view/themeMode` 预置为 `"arctic-frost"`，启动后应用自动选中「冰霜蓝白」且渲染为 Arctic Frost 配色；id 层面向后兼容，用户无感。
- **[T-3]** 点击「冰霜蓝白」「琥珀桃暖」「深夜极光」分别切换，每次切换后再重启应用，主题菜单仍选中同一项；验证 id（而非 name）作为持久化键生效。
- **[T-4]** `tests/core/ThemeLoaderTest.cpp` 若存在断言 `theme.name == "Arctic Frost"` 之类的硬编码字符串需同步更新为中文（扫一遍测试文件确认）；否则测试挂。
- **[T-5]** 用户自定义主题（`%APPDATA%/SimpleMarkdown/themes/*.toml`）中英文 name 均能正常显示，不受本改动影响。

## 风险

- **R1：单元测试断言硬编码**——`tests/core/ThemeLoaderTest.cpp` 若以 `EXPECT_EQ(theme.name, "Arctic Frost")` 之类写法锁定了旧英文名，改动后会直接挂 CI。动作前先 grep 整个 tests/ 目录确认，把相关断言同步改中文（这是方案 A 唯一的隐性成本点）。
- **R2：英文 locale 下观感**——英文系统启动后菜单仍显示中文。若后续收到英文用户反馈，再升级到方案 B（追加 `name_en_US` 字段 + Loader 改造），本次改动不阻塞该演进。
- **R3：i18n 链路误伤**——`lupdate` 不会扫 TOML 文件，所以 `.ts/.qm` 不会有新增条目；但要确保没人在 C++ 中硬编码了 `tr("Arctic Frost")` 作为 fallback（grep 一遍 src/ 确认没有）。现状已确认 MainWindow::rebuildThemeMenu 直接用 `d.displayName`，无 tr 包裹。
- **R4：打包遗漏**——qrc 中 `:/themes/arctic-frost.toml` 是按文件名（id）注册的，本次只改 TOML 内容，文件名/qrc 不变，打包链路无影响；但构建后仍需确认安装包运行时菜单正确显示中文（防止 qm/qrc 缓存问题）。
- **R5：视觉一致性小分歧**——Light / Dark 英文与三款主题中文混排视觉风格不统一。若后续 review 觉得违和，追加一个小 plan 把 Light / Dark 也中文化（"浅色" / "深色"）即可；本 plan 暂不处理以保持改动范围收敛。
