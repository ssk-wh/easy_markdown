# 预览区 WYSIWYG 直接编辑（重磅需求）

> ⚠ **性质**：架构级重磅需求。"完整 WYSIWYG"路径对当前自绘预览是**颠覆性重写**，投入不亚于从头再写一个预览模块。本 plan 的核心产出是：**让主线程先拍板做到哪一级**（路径 C 伪 WYSIWYG / 路径 B 增量光标层 / 路径 A 换 QTextDocument 底座），再投入工程量。
>
> 建议阅读顺序：§2（代价评估）→ §9（决策问题）→ §5（phase 划分）→ 其余章节。

---

## 1. 背景

### 1.1 用户原话
> 预览区支持**直接编辑**（WYSIWYG 所见即所得）：
> - 可以直接在预览渲染出的内容上点击并编辑
> - 支持插入图片（点击→选文件→嵌入）、表格（生成 Markdown 表格语法）、文件链接等
> - 编辑结果实时回写到编辑器的 Markdown 源码
> - 两个方向都能用：编辑器改源码 → 预览渲染（现有）；预览直接编辑 → 源码更新（新）

### 1.2 需求合理性
- **降低心智负担**：用户大多数时间在看渲染后的效果（标题大小、列表缩进、表格格线），让编辑发生在渲染视图是现代笔记/写作类应用的主流体感。
- **富内容插入**：Markdown 对图片、表格、链接的语法虽简单，但在源码里手写 `|---|---|---|` 或算 `![](./assets/xxx.png)` 相对路径仍较繁琐，"点按钮 + 对话框 + 自动生成"是明显的 UX 提升。
- **对标产品**：
  - **Typora**：完全单栏 WYSIWYG，源码语法在用户离开当前行时隐藏；内核是魔改 CodeMirror + 自己的渲染层。
  - **Obsidian Live Preview**：基于 CodeMirror 6 decoration，源码永远在，渲染只叠在 DOM 上。
  - **Notion**：不是 Markdown，是 Block 模型 + rich-text；每个 block 独立可编辑。
  - **Milkdown / Tiptap**：基于 ProseMirror，把 Markdown 当"序列化格式"，内存里是结构化文档树。
  - **VS Code Markdown Preview**：双栏只读预览（接近我们现状），不做 WYSIWYG。

### 1.3 我们的现状定位
当前 SimpleMarkdown 在光谱上处于 "VS Code Markdown Preview" 档次：双栏、左写右看、预览**只读 + 选区 + 复制**。要跳到 Typora 档需要跨越一条很深的鸿沟；跳到 Obsidian Live Preview 档更接近重做一次编辑器。

## 2. 实施代价评估（核心章节）

### 2.1 当前架构简述（必读）
- **完全自绘**：`src/preview/PreviewPainter.cpp`（41 KB）用 `QPainter::drawText` 逐 segment 画到 `QAbstractScrollArea::viewport()` 上，不经过任何 Qt rich-text 栈。
- **单向数据流**：`specs/10-系统架构.md` §3 `[INV-5] AST 不可变`、§5.2 数据流图明确 `Editor → Document → ParseScheduler → AST → PreviewLayout → PreviewPainter`，箭头单向。
- **选区 hit-test 数据**：`PreviewPainter::m_textSegments` 矩形列表（`PreviewWidget.cpp:582 textIndexAtPoint`），只做字符级反查，没有光标概念。
- **已投入的自绘优化**：
  - 高 DPI 度量闭环（全部 `QFontMetricsF` 带 `p->device()`）——Spec `横切关注点/40-高DPI适配.md`
  - 绝对坐标递归（`[INV-COORD-ABS]`）——Spec `横切关注点/20-坐标系统.md`
  - 字体集中化（`PreviewLayout::baseFont()/monoFont()`、禁止 painter 独立构造 QFont）——Spec `模块-preview/03 §3 INV-8/9`
  - inline run 统一基线（2026-04-14 刚修复）——Spec `模块-preview/03 §3 INV-10`
  - 块级缓存 `PreviewBlockCache`、viewport 裁剪、平滑滚动动画、搜索高亮、内容标记

### 2.2 WYSIWYG 所需能力 vs 当前 gap

|------|---------|------------|---------|

| **剪贴板双向** | ✔ 复制 / HTML | 粘贴富文本转 Markdown、剪切 | 🟡 中 |

### 2.3 三条技术路径对比

#### 路径 A：基于 QTextDocument + QTextEdit 重写 Preview
把 `PreviewWidget` 改为 `QTextEdit` 的子类（或封装），让 Qt 的富文本栈接管绘制、光标、选区、IME、undo。预览侧维护一个 `QTextDocument`，由 `PreviewLayout` 转写为 Qt block/frame/table 树。

- **优点**
  - Qt 自带光标、IME、selection、undo、粘贴、拖拽——全部 free
  - 实施最快（相对而言：2-3 个月可出 MVP）
- **代价**
  - **放弃当前自绘栈的全部既有投资**：INV-DPI、INV-COORD-ABS、INV-8/9/10、块缓存、viewport 裁剪、逐字符 segment 等——Spec `模块-preview/02`、`03` **整章作废**，需全部重写 INV-*（改由 QTextDocument 语义定义）
  - **cmark-gfm AST → QTextDocument 桥接层**要从零写。QTextDocument 原生支持 HTML 子集、QTextTable、QTextImage，但：
    - 代码块（fenced，含语言高亮）要自定义 `QTextCharFormat` + 自绘或 syntax highlighter
    - frontmatter 渲染（表格两列 + 原始 YAML 复制）需要 custom `QTextFrame`
    - 数学公式 / 任务列表 / 脚注 / GFM 删除线等 GFM 扩展都要手动映射
    - marking / search highlight 要改用 `ExtraSelection`
  - **性能回退风险**：QTextDocument 对长文档（10k+ 行）的布局性能弱于精调的自绘；目前性能预算（`specs/00 §6`）是 16ms 一帧，QTextDocument 大文档编辑延迟常常 30-80ms
  - **双向 sync 仍要自己写**：QTextEdit 的 `contentsChange(position, charsRemoved, charsAdded)` 给你的是 QTextDocument 坐标，要反向推 Markdown 源码 offset——依旧是本项目最大的自研工作量
  - **主题系统冲击**：当前主题（`plans/归档/2026-04-14-主题插件系统实现.md`）所有颜色直接传给 `QPainter`，换 QTextDocument 后要全部改走 `QTextCharFormat` / stylesheet
- **工程量估算**：**4-6 人月**（单独一个人全职，不含生态回归测试和 bug 修复尾巴）
- **风险等级**：🔴 极高（颠覆已稳定的预览栈，回归面覆盖所有 T-* 测试）

#### 路径 B：在现有自绘基础上增量加编辑器层
保留 `PreviewPainter` / `PreviewLayout`，在 `PreviewWidget` 上新增光标状态机、字符输入协议、选区编辑、IME。**自己写 Qt TextEdit**。

- **优点**
  - 保住所有既有投资（INV-DPI / INV-COORD-ABS / 块缓存 / 字体中心化）
  - 与 Spec 结构兼容——只需要新增 `模块-preview/11-编辑管线.md`
- **代价**
  - **自研 Qt rich-text 栈**：这是 Qt 5 里 QTextEdit 背后**几万行 C++** 的工作量
  - 必须实现的子系统：
    - 光标定位：从 `QPointF` → `LayoutBlock` → block 内 `InlineRun` → run 内字符 offset → 转为编辑器源码 offset
    - 光标渲染：闪烁 timer、绘制于块的绝对坐标（要跟 INV-COORD-ABS 配合）
    - 字符插入/删除：预览侧不直接改 LayoutBlock（违反 INV-5 AST 不可变），而是**把每个按键翻译成编辑器的 Document edit**，然后等新 AST 回来再重画
    - IME 协议：`inputMethodEvent`、`inputMethodQuery(Qt::ImCursorRectangle/ImSurroundingText/...)` 全套
    - 方向键 / Home / End / Shift+方向键 / Ctrl+方向键 / 跨段：全部基于 LayoutBlock 递归 + source-line 映射
    - 组合字符 / emoji / 变体选择符边界
  - **双向 sync 实现**：每次按键先转为 Markdown 语法 edit（"在第 N 行第 M 列插入字符 X"），走 `Document::insert` → ParseScheduler → 新 AST → PreviewLayout。30ms 防抖下用户会看到光标先跳位再回稳。
  - **延迟回流**：在 AST 未回来前，光标位置用什么画？（要么预测、要么"编辑时冻结渲染"）
  - **结构化编辑**（表格、列表层级）全部要自己加快捷键 + 源码 rewrite 逻辑
- **工程量估算**：**8-12 人月**（真正达到"可用"水平），**16+ 人月**（对齐 Typora 水平）
- **风险等级**：🔴 极高（自研 rich-text 是脚下的陷阱田，DPI / IME / 组合字符全是坑）

#### 路径 C：伪 WYSIWYG（推荐 MVP）
**不在预览区做真的编辑**，而是：

1. **预览点击 = 跳到编辑器**：预览点某处 → 计算源码行号/offset → 编辑器 `setCursorPosition` + `ensureVisible` + `setFocus()`。用户打字仍然在编辑器。
2. **插入富内容 = 命令**：工具栏按钮 / Ctrl+Shift+I（图片）/ Ctrl+Shift+T（表格）/ Ctrl+K（链接）→ 弹对话框 → 构造 Markdown 文本 → 以 `Document::insert` 插入到编辑器当前光标位置 → parse → 预览立即显示结果。用户视觉上**看起来像"在预览里插入"**，但实际执行点在编辑器。
3. **回写方向"形式上"成立**：所有插入、结构化操作都走编辑器的 Document edit → undo 栈自动集成。

- **优点**
  - **零架构冲击**：单向数据流（`specs/10 §3 INV-5`）不变、自绘栈不变、所有 INV-* 不变
  - 工程量可控：**2-3 周**出 MVP
  - 用户真正高频痛点（插入图片 / 插入表格）马上解决
  - 与 `plans/2026-04-14-编辑预览目录区域标题条.md`（工具栏）天然协同
- **代价**
  - **不是"真"的 WYSIWYG**：用户在预览区敲键盘依然没反应（焦点会直接跳回编辑器，或者 swallow 掉）——需要产品定位是否能接受
  - 双击预览某单词进入编辑状态的体验不如 Typora 丝滑
- **工程量估算**：**2-3 人周**
- **风险等级**：🟢 低

### 2.4 推荐
**MVP 走路径 C**，先把用户感知最强的"插入图片/表格/链接"和"预览点击跳编辑器"做掉，**把真正的 WYSIWYG 判断推迟到看到 MVP 反馈之后**。如果确实要做真 WYSIWYG，**强烈推荐走路径 A**——Qt 5 时代自己写 rich-text（路径 B）是明知有雷还蹚的选择。

## 3. 插入功能的交互设计（Phase 1 详细）

### 3.1 插入图片
- **入口**：工具栏按钮【图片】(依赖"区域标题条"plan) / 菜单 Insert → Image / 快捷键 `Ctrl+Shift+I` / 右键预览菜单"插入图片"
- **流程**：
  1. `QFileDialog::getOpenFileName(this, tr("选择图片"), lastDir, "Images (*.png *.jpg *.jpeg *.gif *.webp *.svg)")`
  2. 路径策略（**由主线程决策**，见 §9 问题 3）：
     - 方案 a：复制到文档同级 `assets/<docBasename>/` 目录，生成相对路径 `![](assets/xxx/img.png)`
     - 方案 b：内嵌 base64 `![](data:image/png;base64,...)`（适合分享单文件 .md，但图大时 .md 会膨胀）
     - 方案 c：保持原路径不复制，写绝对路径（用户移动文件会断链）
     - **推荐方案 a + 可选 b**（通过设置里 toggle）
  3. 构造文本：`![alt](assets/xxx/img.png)\n` 或带描述 `![图示](...)`
  4. 走 `Document::insert(cursorPos, text)`——复用编辑器 undo
  5. ParseScheduler 防抖触发，预览 30ms 后显示图片
- **Spec 补丁**：`模块-preview/10-Frontmatter渲染.md` 不相关；需新增 `模块-app/XX-资源管理.md` 约束 assets/ 生命周期

### 3.2 插入表格
- **入口**：菜单 Insert → Table / `Ctrl+Shift+T` / 工具栏按钮
- **流程**：
  1. 弹向导对话框 `TableWizard`：行数（default 3）、列数（default 3）、对齐方式（左/中/右/无，每列可选）、是否带表头（default 是）
  2. 生成 Markdown：
     ```
     |  |  |  |
     |---|---|---|
     |  |  |  |
     |  |  |  |
     ```
     （带对齐时 `|:---|:---:|---:|`）
  3. 插入到光标当前行（如果当前行非空，自动前后加空行）
  4. 插入后自动把光标定位到第一个表头单元格内（通过 offset 计算）
- **Spec 补丁**：可不写独立 Spec，挂在 `模块-app` 下

### 3.3 插入文件链接
- **入口**：`Ctrl+K`（业界标准）/ 菜单 Insert → Link
- **流程**：
  1. 对话框：两种模式
     - **URL 模式**：文本框输入 URL + 显示文字（default = URL）
     - **文件模式**：QFileDialog 选文件，自动计算相对路径
  2. 生成 `[显示文字](url)` 或 `[filename.ext](./relative/path.ext)`
  3. 若当前有选区，用选区文本作为"显示文字"占位

### 3.4 预览点击 = 跳编辑器（Phase 1 必做）
- **事件入口**：`PreviewWidget::mousePressEvent`（已有，见 `PreviewWidget.cpp:284`）或双击 `mouseDoubleClickEvent`
- **实现**：
  1. 在现有 `textIndexAtPoint` 之后新增 `sourceOffsetAtPoint(QPointF)`，返回编辑器侧的字符 offset
  2. 定位机制（已具备基础）：
     - `PreviewLayout::yToSourceLine(qreal y)` 已经存在 → 得到 `sourceLine`（0-based 行号）
     - 进一步在 block 内部：用 `LayoutBlock::sourceStartLine` 找到命中的 block → 在 block 的 `inlineRuns` 中按 X 坐标找到命中 run → 用 `QFontMetricsF::horizontalAdvance` 逐字算到字符 → 该字符对应源码哪个 offset
     - 最简方案：只做到行级精度（点哪行 → 光标跳到该行行首 / 行尾）；列级精度作为 stretch goal
  3. 触发方式：双击 preview 某段 → 编辑器 `setCursorPosition(offset)` + `ensureCursorVisible` + `setFocus`
  4. 新增信号 `PreviewWidget::jumpToSourceRequested(int line, int column)`，在 `MainWindow` 里接到后调用 `m_editor->gotoLineColumn(line, column)`

## 5. 推荐 phase 划分

### Phase 1 — 伪 WYSIWYG MVP（路径 C，2-3 周）
**目标**：让用户"在预览里插入图片/表格/链接"的体验成立，点预览某处能跳到编辑器对应位置。

## 8. 风险

1. **真 WYSIWYG 对自绘架构颠覆性**：路径 A 放弃 `specs/模块-preview/02`、`03` 的所有 INV-* 投资；路径 B 自研 Qt rich-text 的工程量堪比再写一次编辑器。主线程要清楚这不是"小优化"而是**新项目级别**的投入。
2. **Qt 5 的 QTextDocument 对 Markdown 渲染支持有限**：GFM 扩展（任务列表、删除线、自动链接、表格对齐）要全部手动桥接；代码块语言高亮无内建，要自己做 QTextCharFormat。
3. **IME 输入协议在自绘 widget 实现复杂度极高**：中日韩用户占比不低，`inputMethodEvent` / `inputMethodQuery` 全协议需覆盖预编辑下划线、候选框位置、组合字符边界、Windows IMM vs macOS NSTextInputClient 差异。路径 B 的最大坑。
4. **Undo/Redo 跨预览↔编辑器双向串接**：路径 C 天然 OK（都走 Document）；路径 B 要保证光标端的 edit 最终必须表达为 Document edit 而非 LayoutBlock edit，否则 undo 栈会撕裂。需新增 INV 强约束。
5. **图片资源生命周期孤儿**：删除 .md 文件不会清 assets/；多文档引用同一图片时重命名/删除会误伤；换目录/另存为需同步搬 assets/——这是独立 plan（`模块-app/XX-资源管理.md`）需要覆盖的范围。
6. **表格 WYSIWYG 编辑（单元格拖宽、增删行列）工程量巨大**：仅"插入"是小工程，但真在预览里编辑表格单元格要求每个 cell 独立光标和选区，单独做一整个 phase。
7. **所见即所得 vs 源码可见性矛盾**：Typora 强隐藏语法（用户看不到 `**`），Notion 是 block 模型完全不给 Markdown，Obsidian 分 Live/Source 模式。我们要走哪条路？——产品定位问题，不是工程问题。
8. **性能预算 16ms 压力**：路径 A 在 10k+ 行文档上 QTextDocument 布局常常 30-80ms；路径 B 光标闪烁每次 paintEvent 都要算光标位置，若不优化会拖累 fps。
9. **DPI 多屏场景**：现有自绘栈已经解决了跨屏 DPI 变化时的坐标一致性（`mousePressEvent` 里检查 `devicePixelRatioF` 并强制 repaint）；路径 A 的 QTextEdit 在多屏 DPR 变化下是否可靠需验证。
10. **主题系统扩展**：路径 A 全盘改走 `QTextCharFormat`，已有 21 个主题字段的语义要整体重映射——非常大的回归测试面。

## 9. 需要主线程决策的问题（必须先回答）

### Q1（最关键）：MVP 就止步于路径 C 伪 WYSIWYG，还是要承诺做到真 WYSIWYG？
- **路径 C only**：Phase 1 交付"插入 3 件套 + 点击跳转"，用户的"预览可编辑"诉求在 80% 场景下解决，工程代价 2-3 周，**强烈推荐**。
- **要做真 WYSIWYG**：必须选路径 A 或 B，投入 4-12 人月，期间预览回归面极大。需要主线程明确优先级——是否值得把这 4-12 人月投进来（而不是其他 feature）？

### Q2（路径分支）：若做真 WYSIWYG，走路径 A（换 QTextDocument 底座）还是路径 B（自研光标层）？
- 路径 A 快但丢弃既有自绘投资；路径 B 留住投资但工程量 2-3 倍。
- 我的建议：**若做，选 A**；路径 B 是"明知山有虎偏向虎山行"。

### Q3（资源策略）：插入图片默认走哪条路径？
- a) 复制到 `<docdir>/assets/<docbase>/` + 相对路径（推荐，便于分享整个目录）
- b) base64 内嵌（单文件 .md 便于分享，但图大时 .md 膨胀）
- c) 保留原绝对路径（最省事，但移动文件断链）

### Q4（Phase 1 功能范围）：3 件套的优先级？
- 默认：图片 > 表格 > 链接（用户最常要手写 Markdown 的顺序）
- 若只能做一个，优先插入图片；这是用户痛点最集中的。

### Q5（快捷键冲突检查）：`Ctrl+Shift+T` 是否与现有"恢复关闭标签页"冲突？`Ctrl+K` 是否与其他搜索快捷键冲突？
- 需要翻一次 `src/app/MainWindow.cpp` 的 QAction 表 + `.claude`/快捷键 Spec 确认
