# 主题插件系统（Theme Plugin System）

## 主线程裁定（2026-04-13）

实施前的 5 个关键决策已锁定：

1. **配置文件格式：TOML**（`toml++` header-only 库，体积可控、注释友好、错误信息对人类好）
2. **MVP 范围：一步到位 80+ 字段**（颜色 / 字体 / 间距 / 边框 / 圆角 / 滚动条 / 分隔线 全覆盖），避免用户主题半年后失效
3. **热重载：V1 不实现**——"重新扫描"菜单足够
4. **主题市场：V1 只预留 schema 字段**，不实现在线下载
5. **内置主题走 loader：是**——内置主题以 TOML 表达，消除"硬编码 vs 配置文件"两份真相

实施时按上述决策直接执行，不再争论。

## 背景

当前主题能力受限严重：
- `Theme::light()` / `Theme::dark()` 在 `src/core/Theme.cpp` 中硬编码
- 用户只能在内置两套之间切换，无法自定义
- 没有配置文件 → 用户无法分享自己的配色
- 覆盖面窄（仅有颜色、无字体/间距/边框/圆角/滚动条）

目标是提供一套**数据驱动**的主题插件系统：
- 主题以文件承载（TOML / YAML / JSON 三选一）
- 覆盖范围从颜色扩展到字体 / 间距 / 边框 / 圆角 / 滚动条 / 分隔线
- 用户可通过菜单导入 / 导出 / 切换
- 内置主题也使用同一套 schema 表达，保证"UI 消费端"只依赖 `Theme` 结构

## 步骤拆分

- [ ] **Step 1：格式选型 + 解析库调研（0.5 天）**
  - 对比 TOML（toml++）/ YAML（yaml-cpp）/ JSON（Qt 自带）三者：二进制增量、错误信息质量、手写体验、注释支持
  - 在 plan 中锁定选型（默认推荐 **TOML**，Qt 外 + header-only + ~200KB）
  - 若选 JSON 则零依赖；若选 TOML 则引入 toml++；若选 YAML 则引入 yaml-cpp（不推荐）
  - 写一份 1 页的选型 ADR（Architecture Decision Record）放到 spec 附近
  - 产出：选型结论、依赖引入方式（submodule / vendored / CMake FetchContent）

- [ ] **Step 2：设计 schema v1.0（1 天）**

- [ ] **Step 3：实现 ThemeLoader 核心（2 天）**

  - 继承机制（extends 链展开 + 环检测 + 深度限制）
  - 非 fail-fast：单字段错误不中止，返回 `ThemeLoadResult` + errors[]
  - 产出：`ThemeLoader.cpp` 实现 + 基础 TDD 单测（每个验证器独立测）

  - 合并算法（base → current 浅覆盖）

  - 单测：A extends B extends builtin:dark；循环 A extends B extends A
  - 产出：继承 + 环检测测例全绿

- [ ] **Step 6：UI 导入/导出 + 菜单（1 天）**

## 风险与回滚

### 风险
1. **解析库引入增加二进制体积**
   - 缓解：选 TOML++（~200KB header-only）或 JSON（0 增量）；避免 yaml-cpp
   - 缓解：release 构建开启 LTO / `-Os` 减少膨胀
2. **内置主题通过 loader 加载导致启动慢**
   - 缓解：内置主题通过 `:/themes/*.toml` Qt 资源系统访问（无磁盘 IO）；加载预算 < 5ms
   - 缓解：加载一次缓存到 `Theme::light()/dark()` 的 static 局部变量
3. **字段覆盖不全导致部分 UI 硬编码色无法通过主题改**
   - 缓解：Step 9 前用 `grep -rn "QColor(\"#" src/` 审计所有硬编码，全部拉进 Theme
   - 缓解：为 T-10 写测试（grep 硬编码数量必须为 0）
4. **继承链引入复杂度（循环 / 过深 / 性能）**
   - 缓解：深度 ≤ 5；visited set 检测环；加载时测一个性能上限（如 ≤ 20ms）
5. **用户主题文件随应用版本升级产生兼容性漂移**
   - 缓解：schema version + 未知字段忽略（向前兼容）+ 迁移文档
6. **TOML 解析库的许可证冲突**
   - 缓解：toml++ 是 MIT（兼容 GPL/LGPL/MIT）；JSON 走 Qt 自带无新依赖
7. **现有 `Theme::light()/dark()` 被其他代码依赖的细微差别**（如用于构造其他 Theme）
   - 缓解：保留函数签名和返回语义，只替换实现
8. **弹窗 / 子对话框未读 Theme 的所有新字段**（违反 30-主题系统 INV-4）
   - 缓解：Step 9 巡检所有 Dialog 源码；对每个 Dialog 验证 setTheme + 颜色 reads-from-Theme

### 回滚方案
- 分阶段合并：Step 1–3 为一个 PR（纯新增 + 单测），Step 4–7 为第二个 PR（开始影响 UI），Step 8–9 为第三个 PR
- 功能开关：`core/FeatureFlags.h` 加 `kEnableThemePlugins`，默认 false；稳定后切 true
- 回滚方式：git revert 相关 commit；或关闭 `kEnableThemePlugins`，`Theme::light()/dark()` 回到硬编码分支
- 数据回滚：用户 themes/ 目录不删除，下次启用仍可用

## 需要主线程决策

以下问题建议主线程或用户在实施前拍板：

1. **格式选型**：TOML++ / JSON / YAML 三选一。推荐 TOML++，次选 JSON（零依赖）。
2. **MVP 范围**：V1 要不要上全部 80+ 字段？还是先上颜色 + 字体（最常用），其他 V2？推荐前者一步到位，避免用户主题文件半年后失效。
3. **热重载**：V1 是否需要 `QFileSystemWatcher` 热重载？推荐**不要**，"重新扫描"按钮足够。
4. **主题市场**：V1 只预留 schema 字段，不实现在线下载，对吗？
5. **内置主题是否走 loader**：推荐"是"（彻底消除硬编码二份真相），但会多一个启动依赖点。如果担心可维持双轨：硬编码 + 加载都生效，以 loader 为准，硬编码仅作 fallback。
