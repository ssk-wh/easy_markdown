# 抛弃 CSS，全面改自绘控件

## 背景

当前 `MainWindow::applyTheme` 依赖一大段 Qt stylesheet 实现主题：
menuBar / tabBar / statusBar / QMenu / QDialog / QMessageBox / QScrollBar / QSplitter / QPushButton……几百行 CSS，深色 / 浅色双分支。

踩过的坑：
- **区域卡片化尝试（2026-04-15）**：给 EditorWidget / PreviewWidget / TocPanel 加 CSS `border + border-radius`，即便补齐 `setFrameShape(StyledPanel)` 和 `WA_StyledBackground` 两个"style hook"仍无法让边框和圆角可见。Qt 对 QAbstractScrollArea 子类的 stylesheet border 绘制路径存在未能查明的限制。
- **深色主题弹窗一致性**：为了让 QDialog / QMessageBox 在深色主题下不出现系统色，硬编码了一整段 QDialog/QMessageBox 的 CSS。既不跟主题（accentColor 除外）、又容易和 Qt 原生样式冲突。
- **不可控**：CSS 的级联 / 选择器优先级 / platform theme 在不同 Qt 版本和 Linux 桌面下行为不一致。

用户明确决定：**全面抛弃 CSS，改用自绘（每个自定义 widget 的 `paintEvent` + 子 widget 替换 Qt 原生控件为自定义自绘版）**。

## 范围（Sprint 级）

### 已经是自绘的，保持
- EditorWidget（自绘）
- PreviewWidget（自绘）
- TocPanel（部分自绘 + 内部 QLabel/QToolButton 依赖 CSS）

### 需要改自绘的 Qt 原生控件

| 控件 | 当前 CSS | 自绘方案 |
|------|---------|---------|
| `QMenuBar` + `QMenuBar::item` | 背景 / hover / 底部 1px 分割线 | 自写 MenuBar widget，paintEvent 画 bg + 每个 menu item + hover state |
| `QMenu`（弹出） | 背景 / item padding / hover / separator / indicator | 自写 PopupMenu（这是最难的一块，涉及 Qt 事件循环 + 全局事件过滤） |
| `QTabBar` | tab 背景 / hover / active 下边框 | 已有 TabBarWithAdd，补齐自绘（目前部分依赖 CSS） |
| `QStatusBar` | 背景 / 顶部 1px / QLabel 子控件 | 自写 StatusBar widget，paintEvent 画 bg + 顶边线 + 遍历子 label |
| `QDialog` / `QMessageBox` | 背景 / QLabel / 按钮 hover / 按钮默认态 | 自写 ThemedDialog 基类 + 自绘按钮（ThemedButton） |
| `QScrollBar` | thumb / track / hover | 自写 ScrollBarWidget，paintEvent 画 track + thumb |
| `QSplitter` handle | 1-2px 宽度 + 背景色 | 已有 SnapSplitter，补齐自绘（目前依赖 CSS） |
| `QPushButton` | 背景 / border / hover / pressed / default | 自写 ThemedButton |

### 其他依赖 CSS 的地方
- 区域卡片化（本 plan 会重做 — 通过 paintEvent 画 border + 圆角 clipRegion）
- 弹窗的主题跟随
- 任何用 `setStyleSheet("color: ...")` 的代码散点

## 动作（分阶段）

### Phase 1 — 基础设施（1-2 周）
- [ ] 新建 `ThemedPaintHelper`：集中统一的绘制函数（`drawCardFrame(painter, rect, theme, radius)` / `drawHoverableButton(painter, rect, state, theme)`），所有自绘控件复用
- [ ] 新建 `ThemedButton`：代替 QPushButton。paintEvent 画背景 / 边框 / hover / pressed / default-indicator / keyboard focus
- [ ] 新建 `ThemedScrollBar`：代替 QScrollBar
- [ ] 新建 `ThemedDialog`：代替 QDialog（基类），收集所有弹窗迁移到它

### Phase 2 — 顶级容器（1 周）
- [ ] 自写 `ThemedMenuBar` 替换 `QMenuBar`（保留 QAction 收集 + 快捷键分发）
- [ ] 自写 `ThemedStatusBar` 替换 `QStatusBar`
- [ ] 自写 `ThemedPopupMenu` 替换 `QMenu`（最复杂，要处理事件穿透 / 二级菜单 / 键盘导航）

### Phase 3 — 迁移与清理（1 周）
- [ ] 把 applyTheme 里的大段 stylesheet **整体删除**，只保留 setStyleSheet("") 清空
- [ ] 每个自绘控件接受 Theme 引用，主题切换时 update()
- [ ] 重做区域卡片化（spec 18）：在容器 widget 的 paintEvent 用 `QPainter::drawRoundedRect` 画卡片边框，不再依赖 widget 自身的 frame 机制
- [ ] 所有 `QMessageBox::*` 调用点迁移到 `ThemedDialog::*`（或自写简化 API）

### Phase 4 — 回归测试（1 周）
- [ ] 8 款内置主题逐一过视觉（Arctic Frost / Sunset Haze / Midnight Aurora / Light / Dark / Paper Mist / Mint Breeze / Monochrome Zen）
- [ ] 高 DPI（125% / 150% / 200%）下所有控件的 border-radius / hover 状态 / 字号
- [ ] 焦点模式、演示模式、多 Tab、对话框栈叠
- [ ] Windows / Linux（X11 + Wayland）双平台

## 风险

1. **工程量巨大**：~4-6 人周，期间会引入大量回归，需要专项测试
2. **菜单 popup 自绘极难**：QMenu 内部涉及 grab mouse / 事件过滤 / 二级菜单定位，自写容易有焦点丢失 / 快捷键不响应 bug
3. **可访问性退步**：Qt 原生控件对屏幕阅读器 / 辅助功能支持好，自绘要自己补
4. **窗口阴影 / DWM 融合**：自绘弹窗在 Windows 下可能丢失 DWM 圆角阴影
5. **IME 兼容**：自写文本输入区域（如果涉及 PopupMenu 里的搜索框）要处理 inputMethodEvent

## 替代方案（轻量化）

如果完全自绘 4-6 周投入不值，可考虑**渐进式**：
- 保留 QMenu / QMenuBar / QStatusBar 的 Qt 原生 + 当前 CSS（它们不是主要痛点）
- 只把"区域卡片化"（本来就是视觉外围）改自绘 → 在 EditorWidget / PreviewWidget / TocPanel 的 paintEvent 最末尾用 QPainter 画一个 roundedRect 外框，覆盖在内容之上（不需要动 frame 机制）
- QDialog / QMessageBox 接受原生样式
- 工作量收敛到 1-2 天

## 验收

- V1（完全自绘）：applyTheme 不再 setStyleSheet；所有主题色变化通过各 widget 的 paintEvent 响应
- V2（渐进式）：至少保证区域卡片化视觉生效（可见 1px 边框 + 6px 圆角）
